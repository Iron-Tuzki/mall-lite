# 商品收藏缓存设计

## 一、目标

商品收藏关系以 MySQL 收藏表作为权威数据源，Redis 作为加速层，主要解决商品详情页和商品列表页频繁判断“是否已收藏”导致的数据库压力。

当前阶段缓存范围：

1. 单个商品是否已收藏。
2. 商品列表批量判断收藏状态。
3. 个人主页最近收藏列表仍从 MySQL 查询，保证排序和商品信息准确。

## 二、Redis Key 设计

### 1. 收藏集合

```text
mall:user:favorites:{userId}
```

类型：`Set`

成员：商品 ID 字符串。

示例：

```text
mall:user:favorites:1 -> 910001, 910002, 910037
```

### 2. 缓存加载标记

```text
mall:user:favorites:loaded:{userId}
```

类型：`String`

作用：区分“用户没有任何收藏”和“缓存还没有加载过”。

如果只使用 Set，当用户收藏为空时 Redis 不会保留空集合，后续每次判断都会回源 MySQL。`loaded` 标记可以避免空收藏用户反复查库。

## 三、读缓存规则

判断收藏状态时：

```text
1. 如果 loaded key 存在：
   直接读取 Redis Set 判断。

2. 如果 loaded key 不存在：
   从 MySQL 查询该用户全部收藏 productId。
   回填 Redis Set。
   设置 loaded key。
   再基于 Redis Set 返回结果。
```

如果 Redis 查询或回填失败：

```text
记录 warn 日志
回退到 MySQL 查询
```

这样 Redis 故障不会影响主链路功能。

## 四、写缓存规则

### 1. 收藏商品

```text
写 MySQL 收藏表
事务提交成功后：
  如果 loaded key 存在，则 Redis SADD productId
  如果 loaded key 不存在，不更新 Redis，保持缓存未加载
```

### 2. 取消收藏

```text
写 MySQL，将收藏记录 deleted=1
事务提交成功后：
  如果 loaded key 存在，则 Redis SREM productId
  如果 loaded key 不存在，不更新 Redis，保持缓存未加载
```

这里不能在 loaded 不存在时直接 `SADD` 并设置 loaded。因为用户可能已经有其他收藏，只是当前缓存还没加载过，直接写入单个商品会把 Redis Set 误标记为完整集合。

## 五、Redis 更新失败处理

如果事务提交后 Redis 增量更新失败：

```text
记录 warn 日志
删除 loaded key
```

删除 loaded 标记后，下次查询会自动回源 MySQL 重建完整 Set，避免 Redis 长时间返回错误收藏状态。

## 六、过期时间

收藏 Set 和 loaded 标记使用相同 TTL：

```text
24 小时 + 0~30 分钟随机抖动
```

目的：

1. 避免 Redis 长期堆积。
2. 避免大量 key 同时过期造成缓存雪崩。
3. 写操作会主动维护已加载缓存，因此 TTL 不需要太短。

## 七、后续扩展

1. 使用 Redis `SMISMEMBER` 或 Lua 优化批量判断。
2. 收藏列表可以缓存商品摘要，但需要处理商品改名、改价、上下架等一致性问题。
3. 收藏变更可以发送 MQ 事件，异步更新缓存和用户画像。
