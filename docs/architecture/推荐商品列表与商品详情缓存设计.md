# 推荐商品列表与商品详情缓存设计

## 一、目标

本阶段补齐前台首页“猜你喜欢”商品列表和商品详情缓存能力，让首页可以从后端分页加载商品摘要，商品详情接口通过 Redis 降低 MySQL 查询压力。

## 二、接口设计

### 1. 推荐商品列表

```http
GET /api/products/recommend?pageNo=1&pageSize=20
```

返回分页商品摘要：

```json
{
  "success": true,
  "data": {
    "pageNo": 1,
    "pageSize": 20,
    "total": 100,
    "records": []
  }
}
```

第一版推荐规则保持简单：

1. 只查询上架商品。
2. 排除逻辑删除商品。
3. 按 `sort` 升序、`id` 倒序返回。
4. `pageNo` 最小为 1。
5. `pageSize` 默认 20，最大 50，避免一次请求过多数据。

### 2. 商品详情

沿用已有接口：

```http
GET /api/products/{productId}
```

缓存 key：

```text
mall:product:detail:{productId}
```

缓存内容为商品详情 JSON，包括商品基础信息和 SKU 列表。

## 三、为什么列表用分页，前端做瀑布流

后端接口采用分页，避免一次查询过多商品。前端展示可以做成类似瀑布流或无限滚动：

1. 第一次请求第一页商品。
2. 用户继续浏览时请求下一页。
3. 前端把下一页数据追加到当前商品列表。

这样用户体验接近瀑布流，但后端仍然保持清晰稳定的分页模型。

## 四、为什么不在列表加载时缓存所有商品详情

列表和详情的数据粒度不同。

商品列表只需要：

1. 商品 ID。
2. 商品名称。
3. 副标题。
4. 主图。
5. 最低价。

商品详情才需要：

1. 商品描述。
2. SKU 列表。
3. 规格信息。
4. 原价和现价。

如果列表一次加载 20 个商品就把 20 个详情都写入 Redis，会带来写入放大、缓存空间浪费和接口变慢。更合适的方式是：列表只查摘要，用户点击商品详情时再缓存详情。

## 五、缓存策略

商品详情缓存策略：

1. 先查 Redis。
2. Redis 命中则直接返回。
3. Redis 未命中则查 MySQL。
4. 查到商品详情后写入 Redis。
5. TTL 默认 30 分钟，并增加 0 到 5 分钟随机值，降低大量 key 同时过期的概率。

当前商品数据主要由 SQL 脚本维护，变化频率低，第一版不做复杂缓存失效。后续引入后台商品管理后，再补商品更新时删除详情缓存。

## 六、前端接入

首页“猜你喜欢”改为调用：

```http
GET /api/products/recommend?pageNo=1&pageSize=20
```

商品卡片点击后进入：

```text
/product/:id
```

详情页调用：

```http
GET /api/products/{productId}
```

如果接口暂时不可用，前端显示加载失败提示，而不是空白页面。

## 七、后续扩展

1. 推荐列表增加分类、关键词、价格区间等筛选条件。
2. 商品列表页增加分页加载或无限滚动。
3. 热门商品列表增加 Redis 缓存。
4. 商品更新后删除详情缓存。
5. 引入浏览记录、订单记录后升级为真正的个性化推荐。
