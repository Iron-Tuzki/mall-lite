# 用户签到与优惠券奖励设计

## 一、设计目标

用户签到功能用于在个人主页展示本月签到记录，并在用户点击“今日签到”后点亮当天记录。签到奖励在此基础上扩展：当用户连续签到达到指定天数时，发放对应优惠券。

当前阶段目标：

1. 支持今日签到，并保证重复点击、请求重试时不会重复处理。
2. 支持查询本月累计签到天数、当前连续签到天数和本月每日签到热力图。
3. 连续签到 7 天，发放一张满 100 减 10 优惠券。
4. 连续签到 15 天，发放一张满 200 减 25 优惠券。
5. 当前采用 RabbitMQ 异步发券，签到接口只负责写签到记录和发送奖励消息。
6. 发券具备幂等能力，重复签到、并发请求或 MQ 重复消费都不能重复发券。

暂不处理年度热力图、积分奖励、补签卡、签到排行榜、下单选择优惠券和订单金额优惠计算，这些能力后续单独扩展。

## 二、Redis 签到数据设计

签到记录使用 Redis Bitmap 按用户、月份存储：

```text
mall:user:sign:{userId}:{yyyyMM}
```

示例：

```text
mall:user:sign:1:202605
```

每日 offset 规则：

```text
offset = dayOfMonth - 1
```

例如：

```text
5月1日  -> offset 0
5月28日 -> offset 27
```

使用 Bitmap 的好处：

1. 单用户单月签到数据占用空间很小。
2. 写入当天签到记录天然具备幂等特性。
3. 统计本月签到天数和构造热力图比较直观。
4. 适合个人主页这种轻量、高频读取场景。

## 三、接口设计

### 1. 今日签到

```text
POST /api/users/sign-in
```

依赖登录态，不需要前端传 `userId`。

响应示例：

```json
{
  "todaySigned": true,
  "monthSignedCount": 14,
  "continuousSignedDays": 7,
  "year": 2026,
  "month": 5,
  "daysInMonth": 31,
  "signedDays": [1, 2, 3, 4, 5, 6, 7]
}
```

核心字段说明：

1. `todaySigned`：今天是否已签到。
2. `monthSignedCount`：本月累计签到天数。
3. `continuousSignedDays`：当前连续签到天数。
4. `signedDays`：本月已签到日期列表，用于个人主页热力图。

当前 `SignInProfileVO` 只返回签到概览，不返回优惠券发放结果。连续签到奖励由 MQ 消费者异步处理，前端可以通过用户优惠券列表查询最终到账结果。

### 2. 查询个人主页签到概览

```text
GET /api/users/sign-in/profile
```

返回签到概览字段，通常不主动触发发券，只用于页面展示。

## 四、签到核心流程

```text
用户点击今日签到
-> 从登录态解析 userId
-> 计算当前月份 Redis Bitmap key 和当天 offset
-> 写入当天签到位，并获取旧值
-> 旧值为 true，说明今天已签到
   -> 直接返回当前签到概览
-> 旧值为 false，说明今天首次签到
   -> 统计本月累计签到天数
   -> 计算当前连续签到天数
   -> 判断是否满足连续签到奖励
   -> 满足门槛时发送优惠券奖励 MQ 消息
   -> 返回签到概览
```

关键点：

1. **只有今天首次签到时才触发奖励检查**，避免重复点击反复进入发券逻辑。
2. 签到记录写入 Redis，优惠券写入 MySQL，二者无法放入同一个本地事务，因此通过 MQ 解耦签到和发券。
3. 查询个人主页概览只读签到状态，不应制造优惠券发放副作用。

## 五、签到幂等设计

Redis Bitmap 写入当天签到位时可以拿到旧值：

1. 旧值为 `false`：说明今天首次签到。
2. 旧值为 `true`：说明今天已经签到，本次请求直接返回当前概览。

这样可以覆盖：

1. 前端按钮重复点击。
2. 前端请求超时后重试。
3. 网络抖动导致重复提交。
4. 用户刷新页面后再次点击签到。

后续如果接积分、成长值或站内信，也应该只在“今天首次签到”这个分支里触发奖励消息。

## 六、连续签到计算

连续签到从当前日期往前倒推：

1. 今天已签到：从今天开始向前统计连续的 `1`。
2. 今天未签到：从昨天开始向前统计连续的 `1`。

这样用户还没签到时，也能看到截至昨天的连续签到天数；用户今日签到后，连续天数会立即增加。

示例：

```text
5月1日、2日、3日已签到，5月4日未签到
-> 5月4日查询概览：continuousSignedDays = 3
-> 5月4日完成签到：continuousSignedDays = 4
```

如果中间断签，则从最近连续区间重新开始计算。

## 七、优惠券奖励规则

当前测试数据包括两个优惠券模板：

| 连续签到天数 | 优惠券模板 ID | 优惠规则 |
| ---: | ---: | --- |
| 7 天 | `700001` | 满 100 减 10 |
| 15 天 | `700002` | 满 200 减 25 |

奖励来源字段设计：

```text
source_type = 1
source_key = SIGN_IN_7_202605
source_key = SIGN_IN_15_202605
```

含义：

1. `source_type = 1` 表示签到奖励。
2. `source_key` 由奖励类型、门槛天数和月份组成。
3. 同一个用户在同一个月份内，连续 7 天和连续 15 天奖励都只能各领取一次。

## 八、优惠券表设计

### 1. 优惠券模板表 `sms_coupon_template`

用于定义优惠券规则，例如优惠券名称、优惠类型、门槛金额、优惠金额、有效期和库存。

关键字段：

1. `id`：优惠券模板 ID。
2. `name`：优惠券名称。
3. `threshold_amount`：使用门槛金额。
4. `discount_amount`：优惠金额。
5. `total_stock`：总库存，可为空表示不限量。
6. `received_count`：已领取数量。
7. `status`：模板状态。

### 2. 用户优惠券表 `sms_user_coupon`

用于记录用户实际获得的优惠券。为了避免重复发券，表中增加唯一索引：

```sql
UNIQUE KEY uk_user_source (user_id, source_type, source_key)
```

这个唯一索引是签到奖励幂等的最终兜底。

## 九、MQ 异步发券流程

```text
今天首次签到成功
-> 统计当前连续签到天数
-> 遍历签到奖励规则
-> 如果 continuousSignedDays >= requiredDays
   -> 构造 CouponRewardMessage
   -> 发送到 mall.coupon-reward.exchange
-> RabbitMQ 路由到 mall.coupon-reward.queue
-> CouponRewardConsumer 消费消息
-> 调用 CouponRewardService.issueSignInContinuousRewards4MQ(...)
-> 生成 source_type 和 source_key
-> 查询用户是否已领取该月该门槛奖励
-> 已领取：跳过
-> 未领取：查询优惠券模板
-> 插入用户优惠券
-> 条件更新模板领取数量
```

当前实现中的关键类：

1. `RedisSignInService`：首次签到成功后，根据连续签到天数发送 `CouponRewardMessage`。
2. `CouponRewardProducer`：实现 `CouponRewardMessageSender`，负责把奖励消息投递到 RabbitMQ。
3. `CouponRewardConsumer`：监听 `mall.coupon.rabbit.coupon-reward-queue`，消费消息并调用发券服务。
4. `CouponRewardServiceImpl`：在事务内完成用户券插入和模板领取数量更新。

这种设计让签到接口尽快返回签到概览，发券逻辑由后台消费者处理。即使消息被重复投递或重复消费，也由用户优惠券唯一索引保证最终不会重复发券。

### 1. 奖励消息内容

`CouponRewardMessage` 包含：

1. `userId`：领取奖励的用户 ID。
2. `continuousSignedDays`：当前连续签到天数。
3. `rewardMonth`：奖励所属月份。
4. `requiredDays`：本次奖励门槛，例如 7 或 15。
5. `templateId`：优惠券模板 ID。

### 2. RabbitMQ 配置

当前优惠券奖励 MQ 配置由 `CouponRewardRabbitProperties` 和 `CouponRabbitConfig` 维护：

| 配置项 | 默认值 | 含义 |
| --- | --- | --- |
| `couponRewardExchange` | `mall.coupon-reward.exchange` | 签到奖励交换机 |
| `couponRewardQueue` | `mall.coupon-reward.queue` | 签到奖励队列 |
| `couponRewardRoutingKey` | `mall.coupon-reward.routing-key` | 签到奖励路由键 |
| `failedExchange` | `mall.coupon.failed.exchange` | 失败消息交换机 |
| `failedQueue` | `mall.coupon.failed.queue` | 失败消息队列 |
| `failedRoutingKey` | `mall.coupon.failed.routing-key` | 失败消息路由键 |
| `timeoutMinutes` | `30` | 奖励消息在队列中的过期时间 |

奖励队列配置了死信交换机和死信路由键，消息在队列中超过 `timeoutMinutes` 未被消费时，会进入失败队列，便于后续排查或补偿。

## 十、发券幂等与并发控制

### 1. 重复签到

Redis Bitmap 写入当天签到位天然幂等。代码中会先判断当天是否已经签到，只有首次签到才会触发发券逻辑。

### 2. 重复消息和重复发券

MQ 可能出现重复投递、消费者重试或业务手动补偿，因此消费侧必须幂等。发券前先根据 `user_id + source_type + source_key` 查询用户是否已经领取奖励：

1. 已领取：直接跳过，不读取模板，不执行插入。
2. 未领取：继续插入用户优惠券。
3. 并发请求同时判断为未领取：由 `uk_user_source` 唯一索引最终兜底。

即使同一条奖励消息被重复消费，也不会重复生成优惠券。

### 3. 库存统计

发券成功后，通过条件更新增加模板 `received_count`：

```sql
UPDATE sms_coupon_template
SET received_count = received_count + 1
WHERE id = ?
  AND status = 1
  AND deleted = 0
  AND (total_stock IS NULL OR received_count < total_stock)
```

如果库存不足，事务回滚用户优惠券插入，避免出现用户券已生成但模板库存没有扣减的情况。

### 4. 惰性补发

发券条件使用 `continuousSignedDays >= requiredDays`，而不是只使用 `==`。

原因是 Redis Bitmap 签到记录、MQ 消息投递和 MySQL 优惠券记录无法放进同一个本地事务。如果用户第 7 天签到成功，但消息发送失败、消息过期进入失败队列，或消费时数据库短暂故障，那么第 8 天签到时仍然应该检查并再次发送 7 天奖励消息，最终由消费侧补发缺失奖励。

惰性补发流程：

```text
用户后续首次签到
-> 当前连续签到天数仍然大于等于奖励门槛
-> 再次发送该月该门槛的奖励消息
-> 消费者查询该月该门槛奖励是否已存在
-> 已存在：直接跳过
-> 不存在：补发优惠券
```

这种方式不会在每次签到时频繁触发唯一索引异常。后续还可以增加失败队列重放、消费失败重试和定时补偿任务。

## 十一、为什么使用 RabbitMQ 而不是 Redis Pub/Sub

Redis Pub/Sub 不适合优惠券发放这类不能丢的业务，因为它不保存消息，也没有 ACK 和失败重试能力。消费者宕机或网络断开时，消息可能直接丢失。

RabbitMQ 更适合当前异步发券链路，因为它支持：

1. 消息持久化。
2. 消费确认。
3. 失败重试。
4. 死信队列。
5. 消费幂等控制。

后续如果发券之外还要增加积分、成长值、站内信、推送等能力，可以继续把“签到奖励”作为统一事件，由不同消费者分别处理。

## 十二、测试重点

1. 首次签到会写入当天 Bitmap。
2. 重复签到不会重复改变签到统计。
3. 本月累计签到天数计算正确。
4. 今日未签到时，连续签到天数从昨天开始统计。
5. 今日签到后，连续签到天数从今天开始统计。
6. 连续 7 天签到会发送 7 天奖励消息，最终只发放一次 7 天奖励。
7. 连续 15 天签到会发送 15 天奖励消息，最终只发放一次 15 天奖励，并且不重复发放 7 天奖励。
8. 重复消费同一奖励消息时，只能生成一张用户优惠券。
9. 优惠券模板库存不足时，不生成用户优惠券。
10. 第 7 天消息发送失败或消费失败后，第 8 天首次签到可以再次发送 7 天奖励消息并补发奖励。
