# 优惠券与签到奖励设计

## 一、目标

在用户签到功能上增加连续签到奖励：

1. 连续签到 7 天，发放一张满 100 减 10 优惠券。
2. 连续签到 15 天，发放一张满 200 减 25 优惠券。
3. 当前阶段先采用同步发券，保证逻辑清晰、便于调试。
4. 后续如果奖励逻辑变重，再升级为 RabbitMQ 异步发券。

## 二、当前阶段为什么先同步发券

签到接口当前只做 Redis Bitmap 记录和统计，整体链路较轻。同步发券的优势是：

1. 实现简单，便于直接观察数据库结果。
2. 调试成本低，不需要先补消息可靠性和消费失败处理。
3. 幂等由数据库唯一索引兜底，重复签到或重复触发不会重复发券。

后续如果发券之外还要增加积分、成长值、站内信、推送等能力，再把“签到奖励”抽成事件，通过 RabbitMQ 异步处理。

## 三、为什么不用 Redis Pub/Sub

Redis Pub/Sub 不适合优惠券发放这类不能丢的业务，因为它不保存消息，也没有 ACK 和失败重试能力。消费者宕机或网络断开时，消息可能直接丢失。

RabbitMQ 更适合后续异步化，因为它支持：

1. 消息持久化。
2. 消费确认。
3. 失败重试。
4. 死信队列。
5. 消费幂等控制。

## 四、核心表设计

### 1. 优惠券模板表 `sms_coupon_template`

用于定义优惠券规则，例如优惠券名称、优惠类型、门槛金额、优惠金额、有效期和库存。

当前测试数据包括：

1. `700001`：连续签到 7 天，满 100 减 10。
2. `700002`：连续签到 15 天，满 200 减 25。

### 2. 用户优惠券表 `sms_user_coupon`

用于记录用户实际获得的优惠券。为了避免重复发券，表中增加唯一索引：

```sql
UNIQUE KEY uk_user_source (user_id, source_type, source_key)
```

签到奖励的来源字段设计：

```text
source_type = 1
source_key = SIGN_IN_7_202605
source_key = SIGN_IN_15_202605
```

这样同一个用户在同一个月份内，连续 7 天和连续 15 天奖励都只能各领取一次。

## 五、同步发券流程

```text
用户点击签到
  -> Redis Bitmap 写入当天签到记录
  -> 统计当前连续签到天数
  -> 如果今天是首次签到
      -> 判断是否满足连续 7 天或 15 天
      -> 查询优惠券模板
      -> 插入用户优惠券
      -> 插入成功后增加模板领取数量
      -> 如果重复插入触发唯一索引，直接视为幂等成功
  -> 返回签到概览
```

## 六、幂等与并发控制

### 1. 重复签到

Redis Bitmap 写入当天签到位天然幂等。代码中会先判断当天是否已经签到，只有首次签到才会触发发券逻辑。

### 2. 重复发券

发券前先根据 `user_id + source_type + source_key` 查询用户是否已经领取奖励：

1. 已领取：直接跳过，不读取模板，不执行插入。
2. 未领取：继续插入用户优惠券。
3. 并发请求同时判断为未领取：由 `uk_user_source` 唯一索引最终兜底。

即使后续改成 MQ，消息重复消费时也不会重复生成优惠券。

### 3. 库存统计

当前发券成功后，会通过条件更新增加模板 `received_count`：

```sql
UPDATE sms_coupon_template
SET received_count = received_count + 1
WHERE id = ?
  AND status = 1
  AND deleted = 0
  AND (total_stock IS NULL OR received_count < total_stock)
```

如果库存不足，事务会回滚用户优惠券插入。

### 4. 惰性补发

发券条件使用 `continuousSignedDays >= requiredDays`，而不是只使用 `==`。

原因是 Redis Bitmap 签到记录和 MySQL 优惠券记录无法放进同一个本地事务。如果用户第 7 天签到成功，但发券时数据库短暂故障，那么第 8 天签到时仍然应该检查并补发缺失的 7 天奖励。

惰性补发流程：

```text
用户后续首次签到
  -> 当前连续签到天数仍然大于等于奖励门槛
  -> 查询该月该门槛奖励是否已存在
  -> 已存在：直接跳过
  -> 不存在：补发优惠券
```

这种方式不会在每次签到时频繁触发唯一索引异常。后续如果发券改为 RabbitMQ 异步消费，还可以增加失败重试和定时补偿任务。
