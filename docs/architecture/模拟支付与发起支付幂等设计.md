# 模拟支付与发起支付幂等设计

## 一、设计目标

当前阶段不接支付宝、微信等真实支付渠道，先用后端模拟第三方支付流程，打通交易主链路：

1. 用户下单后生成待支付订单。
2. 下单时库存从 `available_stock` 转入 `locked_stock`。
3. 用户发起模拟支付，系统创建或返回 `PENDING` 支付流水。
4. 模拟第三方支付通过回调通知成功或失败。
5. 支付成功后，订单状态变为已支付，库存从 `locked_stock` 中确认扣减。
6. 支付失败后，支付流水变为失败，订单仍保持待支付，库存仍保持锁定。

本设计同时解决两个幂等问题：

1. **发起支付幂等**：同一个待支付订单并发发起支付时，只创建一条 `PENDING` 支付流水。
2. **支付回调幂等**：同一笔支付流水重复收到回调时，只允许第一次有效回调推进状态、扣减库存、更新订单。

当前版本不引入支付请求幂等表，优先使用订单行锁、支付流水唯一约束和数据库条件更新完成并发控制。

## 二、接口设计

### 1. 发起支付

```text
POST /api/orders/{orderId}/pay
```

发起支付负责两件事：

1. 模拟调用第三方支付预下单接口。
2. 在本地创建或返回已有的待支付流水。

响应核心字段：

```json
{
  "paymentNo": "P202605250830001234567890",
  "orderId": 1,
  "orderNo": "O202605250829001234567890",
  "orderStatus": 10,
  "paymentStatus": 10,
  "payAmount": 398.00
}
```

字段说明：

1. `orderStatus = 10`：订单仍是待支付。
2. `paymentStatus = 10`：支付流水待回调确认。
3. 同一订单在回调前重复发起支付时，返回已有 `PENDING` 支付流水，不重复创建支付流水。

### 2. 支付回调

```text
POST /api/payments/{paymentNo}/callback
```

请求体：

```json
{
  "mockResult": "SUCCESS"
}
```

`mockResult` 当前支持：

1. `SUCCESS`：模拟第三方支付成功。
2. `FAILED`：模拟第三方支付失败。

## 三、支付流水唯一约束

发起支付的并发问题来自典型的“先查再插”窗口：

```text
请求 A 查询没有 PENDING 流水
请求 B 查询没有 PENDING 流水
请求 A 插入 PENDING 流水
请求 B 插入 PENDING 流水
```

`pms_payment` 原本只有 `payment_no` 唯一索引，无法约束“同一订单只能有一条待支付流水”。MySQL 不支持 PostgreSQL 那种原生部分唯一索引，因此使用生成列表达 `PENDING` 唯一约束：

```sql
pending_order_id BIGINT UNSIGNED GENERATED ALWAYS AS (
    CASE WHEN status = 10 AND deleted = 0 THEN order_id ELSE NULL END
) STORED
```

再增加唯一索引：

```sql
UNIQUE KEY uk_pending_order_channel (pending_order_id, pay_channel)
```

这个约束表达的语义是：

1. 同一订单、同一支付渠道，只能有一条 `PENDING` 支付流水。
2. 支付成功、失败或关闭后，`pending_order_id` 变为 `NULL`，不会阻止历史流水存在。
3. 支付失败后，如果订单仍然待支付，可以重新发起支付并创建新的 `PENDING` 流水。

## 四、发起支付流程

### 1. 核心流程

```text
调用发起支付接口
-> 模拟第三方支付预下单耗时
-> 开启本地事务
-> SELECT ... FOR UPDATE 锁定订单行
-> 校验订单必须是待支付
-> 直接插入 PENDING 支付流水
-> 如果插入成功，返回新支付流水
-> 如果唯一键冲突，使用当前读查询已有 PENDING 流水并返回
-> 提交事务
```

同一订单只有一行 `oms_order`，所以锁住订单行后，同一订单的并发支付请求会串行执行。`PENDING` 唯一约束作为数据库兜底，保证即使未来局部代码绕过行锁，也不能创建多条待支付流水。

### 2. 订单行锁查询

订单 Mapper 增加行锁查询方法：

```text
OrderMapper.selectByIdForUpdate(orderId)
```

SQL：

```sql
SELECT *
FROM oms_order
WHERE id = ?
  AND deleted = 0
FOR UPDATE
```

这个方法必须在事务中调用，否则行锁会立即释放，无法保护后续创建支付流水逻辑。

### 3. 支付事务服务

`PaymentTransactionService.createPendingPayment(orderId)` 中，把普通查询订单改成行锁查询订单：

```text
getActiveOrderForUpdate(orderId)
```

锁住订单后不再先查 `PENDING` 支付流水，而是直接插入支付流水：

```text
插入成功：返回新支付流水
唯一键冲突：查询已有 PENDING 支付流水并返回
```

## 五、支付回调流程

### 1. 成功回调

```text
调用支付回调接口，mockResult = SUCCESS
-> 根据 paymentNo 查询支付流水，拿到 orderId
-> 开启本地事务
-> SELECT ... FOR UPDATE 锁定订单行
-> CAS 更新支付流水：PENDING -> SUCCESS
-> 如果影响行数为 0，使用 SELECT ... FOR UPDATE 读取支付流水最新终态并返回
-> 校验订单必须是待支付
-> 按订单明细扣减 SKU 锁定库存
-> CAS 更新订单：PENDING_PAYMENT -> PAID，并写入 pay_time
-> 如果订单影响行数不是 1，抛出异常并回滚当前事务
-> 提交事务
```

### 2. 失败回调

```text
调用支付回调接口，mockResult = FAILED
-> 根据 paymentNo 查询支付流水，拿到 orderId
-> 开启本地事务
-> SELECT ... FOR UPDATE 锁定订单行
-> CAS 更新支付流水：PENDING -> FAILED
-> 如果影响行数为 0，使用 SELECT ... FOR UPDATE 读取支付流水最新终态并返回
-> 订单保持待支付
-> 库存保持锁定
-> 提交事务
```

### 3. 支付流水 CAS

真实支付渠道可能因为网络抖动、超时重试等原因，多次推送同一笔支付回调。为了避免重复扣库存，支付流水不使用“先查再改”，而是使用条件更新做 CAS 抢占：

```sql
UPDATE pms_payment
SET status = 20,
    pay_time = NOW(),
    callback_content = ?,
    update_time = NOW()
WHERE payment_no = ?
  AND status = 10
  AND deleted = 0;
```

影响行数含义：

1. `affectedRows = 1`：当前请求抢占成功，可以继续扣减锁定库存并更新订单。
2. `affectedRows = 0`：支付流水不存在、已删除，或已经被其他回调处理；当前请求只查询并返回当前状态。

失败回调同理，只是把状态从 `PENDING` 推进为 `FAILED`。

### 4. 订单 CAS

支付成功后，订单状态也要做一次条件更新：

```sql
UPDATE oms_order
SET status = 20,
    pay_time = ?,
    update_time = NOW()
WHERE id = ?
  AND status = 10
  AND deleted = 0;
```

订单 CAS 是第二道防线，避免订单已经被取消、关闭或被其他流程修改后，支付回调仍然把订单强行改成已支付。如果订单 CAS 失败，当前支付回调事务回滚，支付流水成功状态和库存扣减也不会提交。

## 六、锁顺序和隔离级别

发起支付和支付回调都会操作订单与支付流水，两个流程必须保持一致的锁顺序：

```text
订单行锁 -> 支付流水操作
```

锁顺序约定：

1. 发起支付：先 `SELECT ... FOR UPDATE` 锁定订单行，再插入 `PENDING` 支付流水。
2. 支付回调：先根据 `paymentNo` 查询出 `orderId`，再 `SELECT ... FOR UPDATE` 锁定订单行，最后 CAS 更新支付流水。

这样可以避免“回调先持有支付流水锁、发起支付先持有订单锁”导致的相互等待。

另外，MySQL RR 隔离级别下普通 `SELECT` 是快照读。对于“唯一键冲突后查询已有流水”或“CAS 失败后返回当前终态”这类幂等返回逻辑，如果必须读取其他事务刚提交的数据，应使用当前读，例如 `SELECT ... FOR UPDATE`，避免返回旧快照。

## 七、为什么模拟耗时不放在事务里

代码中 `MockThirdPartyPaymentClient` 使用 `Thread.sleep(300)` 模拟第三方支付渠道耗时，这个步骤放在本地数据库事务之前。

原因是：真实生产中调用第三方支付属于外部网络 IO，可能慢、可能超时。如果在数据库事务里等待外部接口，会让数据库连接和行锁持有时间变长，降低系统吞吐，也更容易引发锁等待。

当前实现保持这个边界：

```text
外部耗时操作在事务外
本地状态落库在事务内
```

这个设计能保证本地支付流水不会重复创建，但并发重复请求仍可能重复模拟调用第三方。后续如果要继续优化，可以让本地事务返回“是否新建流水”，只有新建成功时才调用真实第三方预下单接口。

## 八、幂等语义

| 场景 | 处理结果 |
| --- | --- |
| 第一次发起支付 | 创建 `PENDING` 支付流水 |
| 同一订单重复发起支付，已有 `PENDING` 流水 | 返回已有支付流水 |
| 同一订单并发发起支付 | 订单行锁串行化，唯一约束兜底，只创建一条 `PENDING` 支付流水 |
| 支付失败后再次发起支付 | 可以创建新的 `PENDING` 支付流水 |
| 订单已支付 | 返回业务错误 |
| 订单已取消 | 返回业务错误 |
| 重复成功回调 | 返回当前成功状态，不重复扣减库存 |
| 成功后又收到失败回调 | 返回当前终态，不把订单改回待支付或失败 |
| 失败后又收到成功回调 | 返回当前终态，不再推进订单和库存 |

## 九、错误处理

| 场景 | 建议处理 |
| --- | --- |
| 订单不存在 | 返回 `order not found` |
| 订单不是待支付 | 返回订单不可支付的业务错误 |
| 已有 `PENDING` 支付流水 | 返回已有支付流水 |
| 支付流水不存在 | 返回 `payment not found` |
| 回调参数不是 `SUCCESS` 或 `FAILED` | 返回参数错误 |
| 回调命中终态流水 | 返回当前支付流水状态 |
| 支付成功扣减锁定库存失败 | 回滚当前事务，返回库存扣减失败 |
| 支付成功更新订单失败 | 回滚当前事务，避免支付流水和库存提前提交 |

## 十、测试重点

1. 串行重复发起支付只返回一条支付流水。
2. 并发重复发起支付只创建一条支付流水。
3. 并发重复发起支付返回相同 `paymentNo`。
4. 支付成功后再次发起支付会被拒绝。
5. 重复成功回调不会重复扣减 `locked_stock`。
6. 成功回调和失败回调并发到达时，只能有一个结果成为终态。
7. 支付失败后订单仍保持待支付，库存仍保持锁定。
8. 支付失败后再次发起支付可以创建新的 `PENDING` 流水。

## 十一、当前限制和后续扩展

1. 当前支付回调没有签名验签，真实支付渠道必须校验回调来源和参数签名。
2. 当前支付失败后，订单和库存仍保持待支付与锁定状态，后续可以增加重新支付和超时自动取消释放库存。
3. 当前回调结果通过 `mockResult` 手动指定，适合 Swagger 和接口测试；后续可以抽象成支付渠道适配器。
4. 当前没有支付超时关闭逻辑，后续可以结合 RabbitMQ 延迟消息或定时任务实现。
5. 后续如果支付接口增加 `payRequestId`，可以引入 `pms_payment_request` 支付请求幂等表，用于精确记录每一次前端支付请求。
6. 如果接入真实支付渠道，建议补充第三方预支付单号、支付请求日志、支付回调日志、渠道回调唯一号去重和对账任务。
