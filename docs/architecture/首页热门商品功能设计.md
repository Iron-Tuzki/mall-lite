# 首页热门商品功能设计

## 一、目标

为商城首页新增热门商品能力，基于最近 24 小时内的用户行为探测热点商品，并提供独立接口：

```text
GET /api/products/hot?limit=10
GET /api/products/hot/{productId}
```

第一版接入四类正向行为：

| 行为 | 分值 |
|---|---:|
| 浏览商品详情 | 1 |
| 收藏商品 | 4 |
| 加入购物车 | 6 |
| 支付成功 | 10 |

热点统计属于旁路能力。热点模块异常不能影响商品详情、收藏、加购和支付成功等业务主流程。

## 二、方案选择

采用 **Redis 小时桶 + RabbitMQ 异步采集 + 定时聚合**。

每类行为发送统一热点事件。消费者按事件发生时间将分值累加至 Redis 小时桶。定时任务每 30 分钟聚合最近 24 个小时桶，并将结果写入首页热门榜单。首页接口直接读取聚合榜单，榜单为空或 Redis 访问异常时使用现有推荐商品排序兜底。

热门商品详情使用独立接口和独立缓存，避免首页热门流量与普通商品详情缓存完全耦合。该缓存支持缓存空值、随机 TTL 和分布式锁互斥重建缓存，适合作为学习缓存击穿、缓存穿透和热点 Key 保护的实践点。

热门商品详情额外接入 Sentinel，资源名为 `hot-product-detail`。该资源支持接口级 QPS 限流、慢调用比例熔断和统一降级返回，防止热门详情流量在缓存未命中、缓存重建或数据库响应变慢时持续冲击后端。

不采用请求时实时聚合，避免首页流量反复触发 Redis 合并操作。不采用单一 ZSet 累积后统一衰减，保留按小时重算榜单和后续调权能力。

## 三、整体架构

```text
详情浏览、收藏、加购、支付成功
              ↓
     ProductHotEventSender
              ↓
 RabbitMQ 热点交换机、队列、失败队列
              ↓
      ProductHotConsumer
              ↓
 Redis 小时桶 ZSet：商品 ID -> 小时热度分
              ↓
 每 30 分钟 ProductHotAggregationTask
              ↓
Redis 首页榜单 ZSet：商品 ID -> 24 小时加权热度
              ↓
GET /api/products/hot?limit=10
              ↓
查询启用商品与最低 SKU 价格并保持榜单顺序
              ↓
点击热门商品进入 GET /api/products/hot/{productId}
              ↓
Sentinel 资源 hot-product-detail
              ↓
读取热门商品详情独立缓存，缓存未命中时互斥回源重建
```

## 四、Redis 数据设计

### 1. 小时桶

```text
Key: mall:product:hot:hour:yyyyMMddHH
Type: ZSet
Member: 商品 ID
Score: 当前小时累计热度分
TTL: 48 小时
```

小时桶保留 48 小时，但定时任务只聚合最近 24 个桶。额外保留时间用于规避边界过期问题，并支持故障恢复和问题排查。

### 2. 首页榜单

```text
Key: mall:product:hot:homepage
Type: ZSet
Member: 商品 ID
Score: 最近 24 小时加权热度分
TTL: 15 分钟
```

聚合任务先写入临时 Key：

```text
mall:product:hot:homepage:tmp
```

聚合完成后再替换正式榜单，避免首页读取到半成品。

### 3. 浏览去重

```text
Key: mall:product:hot:view:dedup:{actor}:{productId}
TTL: 5 分钟
```

actor 规则：

- 登录用户：`user:{userId}`
- 匿名游客：`device:{X-Device-Id}`

如果请求没有有效登录信息，也没有 `X-Device-Id`，详情接口仍正常返回，但不发送浏览热点事件。

### 4. 消费幂等

```text
Key: mall:product:hot:event:processed:{eventId}
TTL: 48 小时
```

消费者按事件 ID 去重。只有首次处理事件时才累加小时桶分值，避免 RabbitMQ 重复投递导致重复计分。

### 5. 热门商品详情缓存

```text
Key: mall:product:hot:detail:{productId}
Type: String
Value: ProductDetailVO JSON 或空值标记
TTL: 基础 TTL + 随机 TTL
```

热门商品详情不复用普通商品详情缓存，原因是热门商品的访问模型更集中，后续可以单独调大 TTL、做预热、做热点保护，且不影响普通详情接口的缓存策略。

空值缓存：

```text
Value: __NULL__
TTL: 5 分钟
```

当商品不存在时写入空值标记，降低恶意或异常商品 ID 反复穿透数据库的风险。

互斥重建锁：

```text
Key: mall:product:hot:detail:lock:{productId}
Wait: 1 秒
Lease: 10 秒
Retry: 3 次
Retry Interval: 50 毫秒
```

缓存未命中时先尝试获取 Redisson 分布式锁。获取锁成功后再次读取缓存，仍未命中才查询数据库并写入缓存。获取锁失败时短暂等待并轮询缓存，等待期间如果其他请求已经重建完成，则直接返回缓存结果；如果等待后仍未命中，则返回 `503 hot product detail is rebuilding`，不再直接查询数据库，避免热点详情缓存击穿时大量请求压到数据库。

## 五、聚合规则

聚合最近 24 个小时桶，按照商品 ID 加权求和：

```text
商品最终热度 = Σ（小时桶内商品分数 × 当前小时桶权重）
```

第一版采用平滑时间衰减：

```text
权重 = 0.9 ^ 距离当前时间的小时数
```

当前小时权重为 `1.0`，越早的小时桶权重越低。实现使用 Redis `ZUNIONSTORE WEIGHTS ... AGGREGATE SUM`，避免在 Java 内遍历全部商品逐项累加。

定时任务每 30 分钟执行一次，最多保留热度最高的 50 个商品到首页榜单。

## 六、模块与组件

### 1. `mall-product`

| 组件 | 职责 |
|---|---|
| `ProductHotAction` | 定义浏览、收藏、加购、支付成功及其权重 |
| `ProductHotEvent` | 统一热点消息，包含事件 ID、商品 ID、行为类型和发生时间 |
| `ProductHotEventSender` | 消息发送接口，供各业务模块依赖 |
| `ProductHotService` | 浏览去重、小时桶累加、24 小时聚合、热门商品查询 |
| `ProductHotDetailCacheService` | 热门商品详情独立缓存，负责空值缓存、随机 TTL 和互斥重建 |
| `ProductHotProperties` | 配置小时桶、聚合榜单、TTL、榜单数量和窗口大小 |

### 2. `mall-app`

| 组件 | 职责 |
|---|---|
| `ProductHotRabbitProperties` | RabbitMQ 队列配置 |
| `ProductHotRabbitConfig` | 声明热点交换机、队列、失败交换机和失败队列 |
| `ProductHotProducer` | 实现 `ProductHotEventSender`，投递消息 |
| `ProductHotConsumer` | 消费事件并更新 Redis 小时桶 |
| `ProductHotAggregationTask` | 每 30 分钟聚合最近 24 小时桶 |
| `ProductController` | 扩展热门商品列表、热门商品详情接口和匿名设备浏览采集 |

聚合任务复用项目已有 `@RedisDistributedLock`，避免多实例重复聚合。

## 七、行为接入点

| 行为 | 接入位置 | 说明 |
|---|---|---|
| 浏览详情 | `ProductController` | 登录用户优先使用 `userId`，否则读取 `X-Device-Id` |
| 收藏成功 | `ProductFavoriteServiceImpl` | 收藏事务提交后发送事件 |
| 加购成功 | `CartServiceImpl` | 购物车变更消息成功发送后，再发送热点事件 |
| 支付成功 | `PaymentTransactionService` | 支付事务提交后，根据订单明细逐个商品发送事件 |

同一订单中同一商品存在多个 SKU 时，支付成功只按商品维度发送一次热点事件，避免 SKU 拆分导致重复加分。

除浏览去重外，第一版不限制用户对收藏、加购和支付成功行为的计分次数。收藏业务自身保持幂等；支付成功回调自身保持幂等；加购行为按每次成功操作计分。

## 八、接口行为

### 1. 热门商品列表请求

```text
GET /api/products/hot?limit=10
```

`limit` 默认值为 10，最大值为 50。

### 2. 热门商品列表返回逻辑

1. 从正式热门榜单读取前 `limit` 个商品 ID。
2. 批量查询启用商品和 SKU 最低价格。
3. 过滤已下架或已删除商品。
4. 按照 Redis 热度榜单顺序返回商品摘要。
5. 当有效热门商品数量不足时，使用现有推荐商品排序补齐，并排除重复商品。
6. Redis 访问异常时，直接使用现有推荐商品排序兜底。

### 3. 热门商品详情请求

```text
GET /api/products/hot/{productId}
```

热门商品详情接口返回 `ProductDetailVO`，响应结构与普通商品详情接口保持一致，但缓存策略独立。首页热门区域和热门商品列表页跳转详情时，前端使用 `/product/{productId}?source=hot`，商品详情页识别 `source=hot` 后调用热门详情接口。

### 4. 热门商品详情返回逻辑

1. 先读取 `mall:product:hot:detail:{productId}`。
2. 命中商品 JSON 时直接反序列化返回。
3. 命中空值标记时返回商品不存在。
4. 未命中时尝试获取 `mall:product:hot:detail:lock:{productId}` 分布式锁。
5. 获取锁成功后再次检查缓存，仍未命中才查询数据库并写入商品缓存或空值缓存。
6. 获取锁失败时短暂等待并轮询缓存。
7. 等待期间命中缓存时直接返回缓存结果。
8. 等待后仍未命中时返回 `503 hot product detail is rebuilding`，不直接回源数据库。
9. 接口仍复用浏览热点采集逻辑，登录用户按用户维度去重，匿名用户按 `X-Device-Id` 去重。

## 九、异常处理

1. 热点 MQ 发送失败时记录告警日志，业务主流程仍然成功。
2. 消费失败时由 RabbitMQ 重试 3 次，之后进入失败队列。
3. 消费者使用事件 ID 做 Redis 幂等校验。
4. 聚合任务失败时保留旧首页榜单，并记录错误日志，等待下一轮任务重试。
5. 首页读取 Redis 失败或榜单为空时使用推荐商品排序兜底。
6. 浏览去重 Redis 访问失败时跳过浏览热点采集，不影响商品详情返回。
7. 热门商品详情抢锁失败且等待后仍未命中缓存时，返回 `503 hot product detail is rebuilding`，优先保护数据库。
8. 热门商品详情触发 Sentinel QPS 限流时，返回 `429 hot product detail is busy`。
9. 热门商品详情触发 Sentinel 慢调用比例熔断时，返回 `503 hot product detail degraded`。

## 十、Sentinel 保护规则

### 1. 资源定义

```text
Resource: hot-product-detail
Entry: ProductCatalogServiceImpl#getHotProductById
```

热门商品详情使用 `@SentinelResource` 标记服务层资源，而不是只在 Controller 层做 Web 入口限流。这样资源名更贴近业务语义，也方便后续继续拆出 `hot-product-detail-cache-rebuild` 之类的更细粒度资源。

### 2. 默认规则

| 规则 | 默认值 | 说明 |
|---|---:|---|
| QPS 限流 | 50 | 超过阈值后直接返回 429 |
| 慢调用 RT | 500 ms | 单次调用超过该耗时视为慢调用 |
| 慢调用比例 | 0.5 | 统计窗口内慢调用比例达到阈值后触发熔断 |
| 最小请求数 | 5 | 统计窗口内请求数不足时不触发熔断 |
| 统计窗口 | 10000 ms | 熔断判断的统计周期 |
| 熔断时长 | 10 秒 | 熔断打开后短时间内直接降级 |

### 3. Dashboard 连接

应用启动时会把 Spring 配置转换为 Sentinel 系统参数：

```text
project.name=mall-lite
csp.sentinel.dashboard.server=localhost:8858
csp.sentinel.api.port=8719
```

如果本地 Sentinel Dashboard 使用其他端口，只需要调整 `mall.sentinel.dashboard-server`。应用启动并访问一次热门详情后，可以在 Dashboard 中看到 `mall-lite` 应用和 `hot-product-detail` 资源。

## 十二、测试设计

| `ProductHotService` | 同一 actor 在 5 分钟内重复浏览，只发送一次事件 |
| `ProductHotService` | 不同用户或设备浏览同一商品，分别计分 |
| `ProductHotService` | MQ 重复投递同一事件 ID 时，小时桶只累加一次 |
| `ProductHotService` | 行为事件进入对应发生时间的小时桶 |
| `ProductHotService` | 聚合最近 24 个桶时，越近的桶权重越高 |
| `ProductHotService` | 查询热门商品时过滤下架商品并保持热度顺序 |
| `ProductHotDetailCacheService` | 缓存命中时不回源数据库 |
| `ProductHotDetailCacheService` | 商品不存在时写入空值缓存 |
| `ProductHotDetailCacheService` | 获取锁成功后双重检查缓存再回源 |

### 2. RabbitMQ 配置测试

验证热点交换机、热点队列、失败交换机、失败队列和绑定关系均正确声明。

### 3. 接口集成测试

1. `GET /api/products/hot?limit=10` 按热度顺序返回启用商品。
2. 榜单为空时按照现有推荐商品排序兜底。
3. `limit` 为空时默认返回 10 条。
4. `limit` 超过上限时限制为 50 条。
5. 匿名访问详情并传递 `X-Device-Id` 时产生浏览热点事件。
6. 未提供登录信息和设备标识时，详情接口仍正常返回，但不统计浏览热度。
7. `GET /api/products/hot/{productId}` 返回商品详情，并写入独立热门详情缓存。
8. `GET /api/products/hot/{productId}` 触发 Sentinel 限流时返回 429。

## 十三、验收标准

1. 浏览详情、收藏、加购和支付成功均可异步累加到对应小时桶。
2. 浏览热点支持登录用户和匿名设备，5 分钟内防重复计分。
3. 定时任务按时间衰减权重聚合最近 24 小时数据。
4. RabbitMQ 重复消息不会重复加分。
5. `/api/products/hot` 返回热门商品，Redis 异常或空榜时自动兜底。
6. `/api/products/hot/{productId}` 使用独立缓存返回热门商品详情，支持空值缓存、随机 TTL 和分布式锁互斥重建。
7. 首页热门商品和热门商品列表页点击详情时，前端调用热门商品详情接口。
8. 热门商品详情支持 Sentinel 限流、慢调用比例熔断和降级返回。
9. 热点模块异常不影响商品详情、收藏、加购和支付成功链路。
