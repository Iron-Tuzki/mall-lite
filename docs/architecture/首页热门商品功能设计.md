# 首页热门商品功能设计

## 一、目标

为商城首页新增热门商品能力，基于最近 24 小时内的用户行为探测热点商品，并提供独立接口：

```text
GET /api/products/hot?limit=10
```

第一版接入四类正向行为：

| 行为 | 分值 |
|---|---:|
| 浏览商品详情 | 1 |
| 收藏商品 | 4 |
| 加入购物车 | 6 |
| 支付成功 | 10 |

热点统计属于旁路能力。热点模块异常不能影响商品详情、收藏、加购和支付成功等业务主流程。

## 二、方案选择

采用 **Redis 小时桶 + RabbitMQ 异步采集 + 定时聚合**。

每类行为发送统一热点事件。消费者按事件发生时间将分值累加至 Redis 小时桶。定时任务每 5 分钟聚合最近 24 个小时桶，并将结果写入首页热门榜单。首页接口直接读取聚合榜单，榜单为空或 Redis 访问异常时使用现有推荐商品排序兜底。

不采用请求时实时聚合，避免首页流量反复触发 Redis 合并操作。不采用单一 ZSet 累积后统一衰减，保留按小时重算榜单和后续调权能力。

## 三、整体架构

```text
详情浏览、收藏、加购、支付成功
              ↓
     ProductHotEventSender
              ↓
 RabbitMQ 热点交换机、队列、失败队列
              ↓
      ProductHotConsumer
              ↓
 Redis 小时桶 ZSet：商品 ID -> 小时热度分
              ↓
 每 5 分钟 ProductHotAggregationTask
              ↓
 Redis 首页榜单 ZSet：商品 ID -> 24 小时加权热度
              ↓
 GET /api/products/hot?limit=10
              ↓
 查询启用商品与最低 SKU 价格并保持榜单顺序
```

## 四、Redis 数据设计

### 1. 小时桶

```text
Key: mall:product:hot:hour:yyyyMMddHH
Type: ZSet
Member: 商品 ID
Score: 当前小时累计热度分
TTL: 48 小时
```

小时桶保留 48 小时，但定时任务只聚合最近 24 个桶。额外保留时间用于规避边界过期问题，并支持故障恢复和问题排查。

### 2. 首页榜单

```text
Key: mall:product:hot:homepage
Type: ZSet
Member: 商品 ID
Score: 最近 24 小时加权热度分
TTL: 15 分钟
```

聚合任务先写入临时 Key：

```text
mall:product:hot:homepage:tmp
```

聚合完成后再替换正式榜单，避免首页读取到半成品。

### 3. 浏览去重

```text
Key: mall:product:hot:view:dedup:{actor}:{productId}
TTL: 5 分钟
```

actor 规则：

- 登录用户：`user:{userId}`
- 匿名游客：`device:{X-Device-Id}`

如果请求没有有效登录信息，也没有 `X-Device-Id`，详情接口仍正常返回，但不发送浏览热点事件。

### 4. 消费幂等

```text
Key: mall:product:hot:event:processed:{eventId}
TTL: 48 小时
```

消费者按事件 ID 去重。只有首次处理事件时才累加小时桶分值，避免 RabbitMQ 重复投递导致重复计分。

## 五、聚合规则

聚合最近 24 个小时桶，按照商品 ID 加权求和：

```text
商品最终热度 = Σ（小时桶内商品分数 × 当前小时桶权重）
```

第一版采用平滑时间衰减：

```text
权重 = 0.9 ^ 距离当前时间的小时数
```

当前小时权重为 `1.0`，越早的小时桶权重越低。实现使用 Redis `ZUNIONSTORE WEIGHTS ... AGGREGATE SUM`，避免在 Java 内遍历全部商品逐项累加。

定时任务每 5 分钟执行一次，最多保留热度最高的 50 个商品到首页榜单。

## 六、模块与组件

### 1. `mall-product`

| 组件 | 职责 |
|---|---|
| `ProductHotAction` | 定义浏览、收藏、加购、支付成功及其权重 |
| `ProductHotEvent` | 统一热点消息，包含事件 ID、商品 ID、行为类型和发生时间 |
| `ProductHotEventSender` | 消息发送接口，供各业务模块依赖 |
| `ProductHotService` | 浏览去重、小时桶累加、24 小时聚合、热门商品查询 |
| `ProductHotProperties` | 配置小时桶、聚合榜单、TTL、榜单数量和窗口大小 |

### 2. `mall-app`

| 组件 | 职责 |
|---|---|
| `ProductHotRabbitProperties` | RabbitMQ 队列配置 |
| `ProductHotRabbitConfig` | 声明热点交换机、队列、失败交换机和失败队列 |
| `ProductHotProducer` | 实现 `ProductHotEventSender`，投递消息 |
| `ProductHotConsumer` | 消费事件并更新 Redis 小时桶 |
| `ProductHotAggregationTask` | 每 5 分钟聚合最近 24 小时桶 |
| `ProductController` | 扩展热门商品查询接口和匿名设备浏览采集 |

聚合任务复用项目已有 `@RedisDistributedLock`，避免多实例重复聚合。

## 七、行为接入点

| 行为 | 接入位置 | 说明 |
|---|---|---|
| 浏览详情 | `ProductController` | 登录用户优先使用 `userId`，否则读取 `X-Device-Id` |
| 收藏成功 | `ProductFavoriteServiceImpl` | 收藏事务提交后发送事件 |
| 加购成功 | `CartServiceImpl` | 购物车变更消息成功发送后，再发送热点事件 |
| 支付成功 | `PaymentTransactionService` | 支付事务提交后，根据订单明细逐个商品发送事件 |

同一订单中同一商品存在多个 SKU 时，支付成功只按商品维度发送一次热点事件，避免 SKU 拆分导致重复加分。

除浏览去重外，第一版不限制用户对收藏、加购和支付成功行为的计分次数。收藏业务自身保持幂等；支付成功回调自身保持幂等；加购行为按每次成功操作计分。

## 八、接口行为

### 1. 请求

```text
GET /api/products/hot?limit=10
```

`limit` 默认值为 10，最大值为 50。

### 2. 返回逻辑

1. 从正式热门榜单读取前 `limit` 个商品 ID。
2. 批量查询启用商品和 SKU 最低价格。
3. 过滤已下架或已删除商品。
4. 按照 Redis 热度榜单顺序返回商品摘要。
5. 当有效热门商品数量不足时，使用现有推荐商品排序补齐，并排除重复商品。
6. Redis 访问异常时，直接使用现有推荐商品排序兜底。

## 九、异常处理

1. 热点 MQ 发送失败时记录告警日志，业务主流程仍然成功。
2. 消费失败时由 RabbitMQ 重试 3 次，之后进入失败队列。
3. 消费者使用事件 ID 做 Redis 幂等校验。
4. 聚合任务失败时保留旧首页榜单，并记录错误日志，等待下一轮任务重试。
5. 首页读取 Redis 失败或榜单为空时使用推荐商品排序兜底。
6. 浏览去重 Redis 访问失败时跳过浏览热点采集，不影响商品详情返回。

```yaml
    hot:
      hour-key-prefix: "mall:product:hot:hour:"
      homepage-key: "mall:product:hot:homepage"

      bucket-ttl-hours: 48
      homepage-ttl-minutes: 15

      aggregation-limit: 50

      event-exchange: mall.product.hot.event.exchange
      event-queue: mall.product.hot.event.queue
      event-routing-key: mall.product.hot.event.routing-key
      failed-exchange: mall.product.hot.failed.exchange
      failed-queue: mall.product.hot.failed.queue
      failed-routing-key: mall.product.hot.failed.routing-key
```

## 十一、测试设计

### 1. 单元测试

| 测试对象 | 验证内容 |
|---|---|
| `ProductHotService` | 同一 actor 在 5 分钟内重复浏览，只发送一次事件 |
| `ProductHotService` | 不同用户或设备浏览同一商品，分别计分 |
| `ProductHotService` | MQ 重复投递同一事件 ID 时，小时桶只累加一次 |
| `ProductHotService` | 行为事件进入对应发生时间的小时桶 |
| `ProductHotService` | 聚合最近 24 个桶时，越近的桶权重越高 |
| `ProductHotService` | 查询热门商品时过滤下架商品并保持热度顺序 |
| `ProductHotAggregationTask` | 调度时调用聚合服务 |
| `ProductHotConsumer` | 消费热点消息后调用小时桶累加能力 |

### 2. RabbitMQ 配置测试

验证热点交换机、热点队列、失败交换机、失败队列和绑定关系均正确声明。

### 3. 接口集成测试

1. `GET /api/products/hot?limit=10` 按热度顺序返回启用商品。
2. 榜单为空时按照现有推荐商品排序兜底。
3. `limit` 为空时默认返回 10 条。
4. `limit` 超过上限时限制为 50 条。
5. 匿名访问详情并传递 `X-Device-Id` 时产生浏览热点事件。
6. 未提供登录信息和设备标识时，详情接口仍正常返回，但不统计浏览热度。

## 十二、验收标准

1. 浏览详情、收藏、加购和支付成功均可异步累加到对应小时桶。
2. 浏览热点支持登录用户和匿名设备，5 分钟内防重复计分。
3. 定时任务按时间衰减权重聚合最近 24 小时数据。
4. RabbitMQ 重复消息不会重复加分。
5. `/api/products/hot` 返回热门商品，Redis 异常或空榜时自动兜底。
6. 热点模块异常不影响商品详情、收藏、加购和支付成功链路。
