# 商品浏览足迹设计

## 一、背景与目标

个人主页已经预留“浏览足迹”区域。第一期实现面向登录用户的近期浏览记录，使用 Redis ZSet 保存最近访问过的商品，不新增 MySQL 表。

目标：

1. 登录用户访问商品详情页时记录浏览足迹。
2. 匿名用户访问商品详情页时正常返回商品信息，但不记录足迹。
3. 同一商品重复浏览时仅更新时间，不重复占位。
4. 个人主页展示最近 3 条足迹。
5. 独立足迹页展示最近 100 条足迹，并支持删除单条和清空全部。
6. 足迹能力异常时不能影响商品详情查询。

## 二、第一期存储方案

### 1. Redis 数据结构

每个用户使用一个 Redis ZSet：

```text
key: mall:user:footprints:{userId}
member: productId
score: 浏览时间戳，单位为毫秒
```

示例：

```text
ZADD mall:user:footprints:1 1780210800000 910037
```

选择 ZSet 的原因：

1. `member` 使用商品 ID，可以天然保证同一商品不重复。
2. 再次浏览相同商品时，`ZADD` 会更新 score，使商品重新回到列表顶部。
3. 可以根据 score 倒序读取最近浏览商品。
4. 可以按排名删除旧数据，只保留最近 100 条。
5. 可以为整个用户足迹 key 设置过期时间。

### 2. 容量与过期时间

第一期约束：

```text
每个用户最多保留：100 条
Redis key 过期时间：90 天
```

每次成功记录足迹后，重新设置 90 天 TTL。用户持续浏览时，近期足迹会持续保留；长期未访问时，Redis 自动释放空间。

### 3. 商品信息查询

ZSet 只保存商品 ID 和浏览时间，不保存商品标题、图片、价格等快照。

读取足迹时：

1. 从 Redis ZSet 按 score 倒序读取商品 ID 和浏览时间。
2. 批量查询 `pms_product` 和对应 SKU 最低价格。
3. 按 Redis 返回的商品 ID 顺序组装响应。

这样可以保证商品信息修改后，足迹页面展示最新的标题、图片和价格。

## 三、写入流程

### 1. 商品详情查询

用户访问：

```text
GET /api/products/{productId}
```

处理流程：

1. 查询商品详情。
2. 商品不存在时沿用现有异常处理，不写入足迹。
3. 商品存在时，从请求头尝试解析登录 token。
4. token 有效时，记录当前用户的商品浏览足迹。
5. 未携带 token 或 token 已失效时，跳过足迹写入。
6. 返回商品详情。

### 2. 可用性原则

足迹属于辅助能力，不能阻断商品详情查询。

因此：

1. 匿名访问商品详情是合法场景。
2. token 失效时不因为足迹写入返回 `401`。
3. Redis 写入异常时记录日志，但仍返回商品详情。

## 四、接口设计

足迹管理接口要求用户已登录。请求头统一使用：

```text
Authorization: Bearer {token}
```

### 1. 查询浏览足迹

```text
GET /api/product-footprints?limit=3
```

参数：

| 参数 | 是否必填 | 说明 |
| --- | --- | --- |
| `limit` | 否 | 查询条数。默认返回最近 100 条，最大值为 100 |

个人主页传入 `limit=3`，完整足迹页不传参数。

响应元素：

| 字段 | 说明 |
| --- | --- |
| `productId` | 商品 ID |
| `name` | 商品名称 |
| `subtitle` | 商品副标题 |
| `mainImageUrl` | 商品主图 |
| `minPrice` | SKU 最低价格 |
| `browseTime` | 最近一次浏览时间 |

### 2. 删除单条足迹

```text
DELETE /api/product-footprints/{productId}
```

语义：

1. 删除当前登录用户指定商品的足迹。
2. 商品不存在于足迹时保持幂等，仍然返回成功。

### 3. 清空全部足迹

```text
DELETE /api/product-footprints
```

语义：

1. 删除当前登录用户的足迹 ZSet。
2. 用户没有足迹时保持幂等，仍然返回成功。

## 六、异常处理

| 场景 | 行为 |
| --- | --- |
| 匿名访问商品详情 | 正常返回详情，不记录足迹 |
| token 失效后访问商品详情 | 正常返回详情，不记录足迹 |
| Redis 写入足迹失败 | 记录日志，正常返回详情 |
| 未登录访问足迹管理接口 | 返回 `401` |
| 删除不存在的足迹 | 幂等返回成功 |
| Redis 中商品已下架或已删除 | 查询列表时过滤该商品 |

## 七、测试范围

### 1. 后端

1. 首次浏览商品后，ZSet 中新增商品 ID。
2. 重复浏览同一商品后，ZSet 数量不增加，score 更新。
3. 浏览超过 100 个商品后，仅保留最近 100 条。
4. 查询足迹时按照最近浏览时间倒序返回。
5. 删除单条足迹保持幂等。
6. 清空全部足迹保持幂等。
7. 匿名访问商品详情时不写入足迹。
8. token 失效时不影响商品详情返回。

### 2. 前端

1. 个人主页展示最近 3 条真实足迹。
2. 点击“全部足迹”进入 `/footprints`。
3. 足迹页可以打开商品详情。
4. 删除单条和清空全部后，页面及时更新。
5. 没有足迹时展示空状态。

## 八、Redis 持久化学习方案

浏览足迹第一期只保存近期数据，业务上允许少量丢失，因此适合作为 Redis 持久化的低风险练习场景。本方案的学习目标不是把浏览足迹升级为强可靠数据，而是通过真实 ZSet 写入观察 RDB、AOF 和混合持久化的恢复行为。

### 1. 学习目标

1. 理解 RDB 快照如何保存足迹 ZSet 的某一时刻状态。
2. 理解 AOF 如何通过重放 `ZADD`、`EXPIRE`、`ZREM`、`DEL` 等写命令恢复足迹。
3. 对比 RDB 与 AOF 在恢复速度、文件大小和数据丢失窗口上的差异。
4. 验证 Redis key 的 TTL 在持久化恢复后是否仍然保留。
5. 明确 Redis 持久化只能提升 Redis 自身恢复能力，不能替代 MySQL 级别的业务持久化。

### 2. 推荐练习配置

本地学习时建议准备三组配置分别验证。

#### (1) RDB 快照模式

```text
appendonly no
save 60 1
```

验证重点：

1. 写入浏览足迹后等待自动快照，或手动执行 `BGSAVE`。
2. 重启 Redis 后检查 `mall:user:footprints:{userId}` 是否恢复。
3. 在快照生成后继续写入新足迹，再模拟 Redis 异常停止，观察最近写入是否可能丢失。

#### (2) AOF everysec 模式

```text
appendonly yes
appendfsync everysec
```

验证重点：

1. 连续浏览多个商品，观察 AOF 记录足迹写命令。
2. 重启 Redis 后检查 ZSet 顺序、score 和 TTL 是否恢复。
3. 理解 `everysec` 通常最多存在约 1 秒数据丢失窗口，性能和可靠性较均衡。

#### (3) RDB + AOF 混合持久化

```text
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
```

验证重点：

1. 理解 AOF 文件前半部分可以使用 RDB 格式保存基础快照。
2. Redis 恢复时先加载快照，再重放增量写命令。
3. 对比纯 AOF 的恢复耗时和文件体积。

### 3. 验证操作建议

可以使用一个固定测试用户进行验证：

```text
mall:user:footprints:1
```

建议流程：

1. 登录后依次访问多个商品详情页，产生浏览足迹。
2. 使用 Redis 客户端查看 ZSet：

```text
ZREVRANGE mall:user:footprints:1 0 -1 WITHSCORES
TTL mall:user:footprints:1
```

3. 根据当前练习模式触发 `BGSAVE`、等待 AOF 刷盘或执行 Redis 重启。
4. 重启后再次执行 `ZREVRANGE` 和 `TTL`，确认足迹数据和剩余过期时间。
5. 删除单条足迹或清空全部足迹后重启 Redis，确认删除命令也会被持久化恢复。

### 4. 业务边界

浏览足迹属于辅助体验数据，第一期不新增 MySQL 表，Redis 是唯一存储。即使开启 RDB 或 AOF，也接受以下情况：

1. Redis 异常停止导致最近少量足迹丢失。
2. 用户长期未访问后，足迹 key 因 TTL 到期自动删除。
3. Redis 数据损坏或被清理后，浏览足迹无法从 MySQL 回源恢复。

如果后续要将浏览历史用于用户画像、推荐系统或运营分析，应演进为 Redis + RabbitMQ + MySQL，由 MySQL 保存长期历史，Redis 只承担近期查询和展示。

## 九、后续演进

第一期仅使用 Redis 保存近期足迹。后续如果要支持长期历史、用户画像和推荐系统，可以演进为 Redis + MySQL：

1. 商品浏览时仍然同步更新 Redis ZSet，保证近期查询性能。
2. 同时将浏览事件异步投递到 RabbitMQ。
3. 消费者批量写入 MySQL 浏览历史表，降低商品详情请求的写库压力。
4. MySQL 保存长期历史，用于用户画像、推荐排序和运营分析。
5. Redis 继续只保留近期热点足迹，作为个人主页和足迹页的快速查询来源。
