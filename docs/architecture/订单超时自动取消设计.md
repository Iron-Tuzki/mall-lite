# 订单超时自动取消设计

## 一、设计目标

订单超时自动取消用于处理用户下单后长时间未支付的场景。当前项目已经具备下单、锁库存、手动取消订单、模拟支付和支付回调能力，下一步需要引入 RabbitMQ，把“待支付订单超过指定时间后自动取消并释放库存”的能力异步化。

第一版目标如下：

1. 下单成功后发送一条订单超时检查消息。
2. 消息延迟一段时间后被消费，例如 15 分钟或 30 分钟。
3. 消费者只取消仍处于待支付状态的订单。
4. 自动取消订单时释放已锁定库存。
5. 支持消费端幂等，避免重复取消订单、重复释放库存。
6. 支持和手动取消、支付成功回调并发时保持状态一致。

第一版暂不处理：

1. 分布式事务最终一致性方案。
2. 消息表、事务消息、Outbox 模式。
3. RabbitMQ 延迟消息插件。
4. 订单关闭原因、取消类型等精细化运营字段。

## 二、方案选择

### 1. 推荐方案：TTL + 死信队列

RabbitMQ 3.12.0 默认支持 TTL 和死信队列，因此第一版采用“延迟队列 + 死信队列”的方式实现订单超时取消。

核心流程如下：

```text
下单成功
  -> 发送消息到订单延迟队列
  -> 消息在延迟队列中等待 TTL 时间
  -> TTL 到期后消息进入死信交换机
  -> 死信交换机路由到订单取消队列
  -> 消费者检查订单状态并尝试取消订单
```

### 2. 暂不使用 RabbitMQ 延迟消息插件

RabbitMQ 的 `x-delayed-message` 插件可以更直观地实现延迟消息，但需要额外安装和启用插件。当前项目以学习核心链路为主，优先使用 RabbitMQ 原生能力，降低本机环境和部署复杂度。

### 3. 暂不用定时任务扫描

定时任务扫描数据库也能完成超时取消，但需要定期查询订单表，后续订单量上来后会带来额外扫描压力。RabbitMQ 更适合当前阶段学习异步消息、延迟处理和消费幂等。

## 三、RabbitMQ 组件设计

第一版建议定义以下交换机、队列和路由键：

| 类型 | 名称 | 作用 |
| --- | --- | --- |
| 延迟交换机 | `mall.order.delay.exchange` | 接收下单成功后的超时检查消息 |
| 延迟队列 | `mall.order.delay.queue` | 持有消息，等待 TTL 到期 |
| 延迟路由键 | `mall.order.delay.routing-key` | 将消息路由到延迟队列 |
| 取消交换机 | `mall.order.cancel.exchange` | 接收从延迟队列转发出来的死信消息 |
| 取消队列 | `mall.order.cancel.queue` | 订单超时取消消费者监听的队列 |
| 取消路由键 | `mall.order.cancel.routing-key` | 将死信消息路由到取消队列 |

延迟队列需要配置死信队列：

```text
x-dead-letter-exchange = mall.order.cancel.exchange
x-dead-letter-routing-key = mall.order.cancel.routing-key
```

TTL 可以先通过配置项控制，例如：

```yaml
mall:
  order:
    timeout-minutes: 30
```

## 四、消息模型设计

订单超时检查消息建议保持简单，只放消费者重新查询订单所需的关键字段：

```json
{
  "orderId": 1,
  "orderNo": "O202605250829001234567890"
}
```

字段说明：

1. `orderId`：订单主键，消费者根据它查询订单。
2. `orderNo`：订单号，用于日志排查和消息内容可读性。

不建议在消息里放订单金额、用户地址、商品明细等完整快照。因为超时取消以数据库当前状态为准，消息只负责触发一次检查。

## 五、下单后的消息发送时机

订单超时消息必须在下单事务提交成功后再发送。

原因是：如果在事务提交前发送消息，而数据库事务随后回滚，就可能出现 RabbitMQ 里有订单超时消息，但数据库里没有对应订单的情况。

第一版可以使用 Spring 的事务同步能力：

```text
创建订单事务提交成功
  -> afterCommit 回调
  -> 发送订单超时检查消息
```

后续如果要进一步提高可靠性，可以演进为 Outbox 本地消息表方案。

## 六、消费者处理流程

消费者收到超时消息后，不直接取消订单，而是先查询订单当前状态。

```text
收到订单超时消息
  -> 根据 orderId 查询订单
  -> 订单不存在：直接确认消息
  -> 订单已支付：直接确认消息
  -> 订单已取消：直接确认消息
  -> 订单待支付：尝试取消订单
      -> CAS 更新订单状态
      -> 更新成功后释放库存
      -> 更新失败说明被其他流程抢先处理，直接结束
```

这里的核心规则是：

```text
只有 PENDING_PAYMENT 状态的订单可以被超时自动取消。
```

## 七、幂等与并发控制

订单超时取消必须和以下场景并发安全：

1. 用户手动取消订单。
2. 支付成功回调更新订单为已支付。
3. RabbitMQ 消息重复投递。
4. 多个消费者并发消费同一条或同一业务含义的消息。

因此自动取消订单不能依赖“先查询再判断”的普通逻辑，而要使用数据库条件更新做 CAS：

```sql
UPDATE oms_order
SET status = 30,
    cancel_time = NOW(),
    update_time = NOW()
WHERE id = ?
  AND status = 10
  AND deleted = 0;
```

影响行数的含义：

1. `affectedRows = 1`：当前线程抢占成功，可以释放库存。
2. `affectedRows = 0`：订单不存在、已取消、已支付，或者已被其他线程处理，当前消费者不再释放库存。

这样可以保证手动取消和超时取消同时发生时，只有一个流程真正取消订单并释放库存。

## 八、代码边界设计

### 1. 订单模块

订单模块负责业务编排：

```text
OrderService.createOrder(...)
  -> 创建订单
  -> 注册事务提交后发送超时消息

OrderService.cancelTimeoutOrder(...)
  -> 查询订单
  -> CAS 取消待支付订单
  -> 释放库存
```

建议把手动取消和超时取消的核心能力收口到同一个内部方法中，例如：

```text
cancelPendingOrder(orderId, cancelSource)
```

这样可以避免两套取消逻辑长期分叉。

### 2. 库存模块

库存模块继续只提供库存能力：

```text
InventoryService.releaseStock(...)
```

它不关心订单是用户手动取消，还是 MQ 超时取消。

### 3. MQ 适配层

MQ 相关代码建议放在 `mall-app` 或订单模块的消息适配包中：

```text
OrderRabbitProperties
OrderRabbitConfig
OrderTimeoutMessage
OrderTimeoutProducer
OrderTimeoutConsumer
```

第一版如果为了让配置和监听更贴近启动应用，可以先放在 `mall-app`；如果后续订单模块要独立成服务，再整体迁移到 `mall-order` 对应服务中。

### 4. 当前已落地的消息发送边界

当前代码已经先落地消息体和生产者：

```text
mall-order:
  OrderTimeoutMessage
  OrderTimeoutMessageSender

mall-app:
  OrderTimeoutProducer
```

`OrderService.createOrder(...)` 在订单创建成功、订单明细落库、幂等请求状态标记成功后，注册 Spring 事务同步回调：

```text
事务提交成功
  -> afterCommit
  -> OrderTimeoutMessageSender.send(...)
```

这样可以避免事务回滚但 MQ 已经收到超时消息的问题。当前第一版没有引入本地消息表，因此 `afterCommit` 中发送 MQ 失败时只记录日志，不反向影响已经提交成功的订单。后续如果要提高可靠性，再演进为 Outbox 本地消息表。

## 九、异常处理与重试策略

第一版建议使用 RabbitMQ 默认重试能力配合日志记录：

1. 业务幂等场景直接正常返回，让消息确认。
2. 数据库临时异常、连接异常可以抛出异常，让监听容器按配置重试。
3. 重试多次仍失败的消息，后续可以接入失败队列或人工排查日志。

消费者里不要吞掉未知异常，否则消息会被错误确认，导致订单可能永远不会被自动取消。

## 十、测试重点

第一版测试重点不是强依赖真实 RabbitMQ，而是优先验证业务逻辑正确：

1. 待支付订单收到超时消息后会被取消。
2. 已支付订单收到超时消息后不会被取消。
3. 已取消订单重复收到消息不会重复释放库存。
4. 手动取消和超时取消并发时，只会有一个流程释放库存。
5. 下单事务回滚时不发送超时消息。

如果本机 RabbitMQ 环境稳定，再补充真实消息发送和消费的集成测试。

## 十一、后续演进方向

1. 增加订单取消原因，例如 `TIMEOUT_CANCEL`、`USER_CANCEL`。
2. 增加本地消息表，解决事务提交后发送 MQ 失败的问题。
3. 增加失败队列，方便排查反复消费失败的消息。
4. 使用 RabbitMQ 延迟消息插件，让不同订单支持不同延迟时间。
5. 拆分订单服务后，把超时取消消费者迁移到独立订单服务中。
