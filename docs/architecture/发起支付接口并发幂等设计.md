# 发起支付接口并发幂等设计

## 一、设计目标

发起支付接口用于把待支付订单转换为一条本地待支付流水，并模拟调用第三方支付渠道。

当前接口是：

```text
POST /api/orders/{orderId}/pay
```

本次优化目标是解决同一个订单被并发点击支付时可能创建多条 `PENDING` 支付流水的问题。

第一版目标如下：

1. 同一个待支付订单并发发起支付时，只创建一条待支付流水。
2. 重复发起支付时，返回已有待支付流水。
3. 支付成功后的订单不能再次发起支付。
4. 不引入支付请求幂等表，优先使用支付流水唯一约束作为请求抢占边界。

## 二、当前问题

当前发起支付逻辑是：

```text
查询订单
-> 校验订单是否可支付
-> 查询是否已有 PENDING 支付流水
-> 如果有，直接返回
-> 如果没有，创建 PENDING 支付流水
```

串行请求没有问题，但并发请求存在窗口：

```text
请求 A 查询没有 PENDING 流水
请求 B 查询没有 PENDING 流水
请求 A 插入 PENDING 流水
请求 B 插入 PENDING 流水
```

因为 `pms_payment` 当前只有 `payment_no` 唯一索引，没有约束“同一订单只能有一条待支付流水”，所以需要补并发控制。

## 三、方案选择

### 1. 推荐方案：订单行锁 + PENDING 流水唯一约束

当前采用 `SELECT ... FOR UPDATE` 锁订单主表行，并在支付流水表增加只针对 `PENDING` 流水生效的唯一约束。

核心流程：

```text
开启本地事务
-> select oms_order where id = ? for update
-> 校验订单状态是否允许支付
-> 直接插入 PENDING 支付流水
-> 如果插入成功，返回新流水
-> 如果唯一键冲突，查询并返回已有 PENDING 流水
-> 提交事务
```

因为同一个订单只有一行 `oms_order`，所以锁住订单行后，**同一订单的并发支付请求会串行执行**。唯一约束作为数据库兜底，保证即使未来局部代码绕过行锁，也不能创建多条待支付流水。

### 2. 暂不使用支付幂等表

支付请求幂等表更适合接口具备 `requestId` 的场景，例如：

```json
{
  "payRequestId": "xxx"
}
```

当前支付接口只有 `orderId`，没有前端支付请求幂等号。此时用订单行锁更简单，也更符合当前项目阶段。

### 3. PENDING 流水唯一约束

MySQL 不支持 PostgreSQL 那种原生部分唯一索引，不能直接写：

```text
同一 order_id 下 status = PENDING 的记录唯一
```

因此使用生成列 `pending_order_id`：

```sql
pending_order_id BIGINT UNSIGNED GENERATED ALWAYS AS (
    CASE WHEN status = 10 AND deleted = 0 THEN order_id ELSE NULL END
) STORED
```

再增加唯一索引：

```sql
UNIQUE KEY uk_pending_order_channel (pending_order_id, pay_channel)
```

这样可以表达：

1. 同一订单、同一支付渠道，只能有一条 `PENDING` 支付流水。
2. 支付成功或失败后，`pending_order_id` 变为 `NULL`，不会阻止后续历史流水存在。
3. 支付失败后，如果订单仍然待支付，可以重新发起支付并创建新的 `PENDING` 流水。

## 四、实现细节

### 1. 订单 Mapper 增加行锁查询

新增方法：

```text
OrderMapper.selectByIdForUpdate(orderId)
```

SQL：

```sql
SELECT *
FROM oms_order
WHERE id = ?
  AND deleted = 0
FOR UPDATE
```

这个方法必须在事务中调用，否则行锁会立即释放，无法保护后续创建支付流水逻辑。

### 2. 支付事务服务改造

`PaymentTransactionService.createPendingPayment(orderId)` 中，把普通查询订单改成行锁查询订单：

```text
getActiveOrderForUpdate(orderId)
```

锁住订单后不再先查 `PENDING` 支付流水，而是直接插入支付流水：

```text
插入成功：返回新支付流水
唯一键冲突：查询已有 PENDING 支付流水并返回
```

### 3. 模拟第三方支付调用

当前模拟第三方支付调用在本地事务外执行，避免数据库事务长期占用连接。

第一版仍保持这个边界不变：

```text
模拟第三方支付耗时
-> 本地事务内创建或返回支付流水
```

它能保证本地支付流水不会重复创建，但并发重复请求仍可能重复模拟调用第三方。后续如果要继续优化，可以让本地事务返回“是否新建流水”，只有新建成功时才调用第三方。

## 五、幂等语义

| 场景 | 处理结果 |
| --- | --- |
| 第一次发起支付 | 创建 `PENDING` 支付流水 |
| 同一订单重复发起支付，已有 `PENDING` 流水 | 返回已有支付流水 |
| 同一订单并发发起支付 | 唯一约束兜底，只创建一条 `PENDING` 支付流水 |
| 支付失败后再次发起支付 | 可以创建新的 `PENDING` 支付流水 |
| 订单已支付 | 返回业务错误 |
| 订单已取消 | 返回业务错误 |

## 六、测试重点

1. 串行重复发起支付只返回一条支付流水。
2. 并发重复发起支付只创建一条支付流水。
3. 并发重复发起支付返回相同 `paymentNo`。
4. 支付成功后再次发起支付会被拒绝。

## 七、后续演进

后续如果支付接口增加 `payRequestId`，可以再引入 `pms_payment_request` 支付请求幂等表，用于精确记录每一次前端支付请求。

如果接入真实支付渠道，也建议补充：

1. 第三方预支付单号。
2. 支付请求日志。
3. 支付回调日志。
4. 渠道回调唯一号去重。

## 八、与支付回调的锁顺序约定

发起支付和支付回调都可能操作同一订单与同一组支付流水，因此两个流程必须保持一致的锁顺序：

```text
订单行锁 -> 支付流水操作
```

发起支付已经先通过 `SELECT ... FOR UPDATE` 锁定订单，再插入 `PENDING` 支付流水。支付回调也需要先根据 `paymentNo` 找到 `orderId`，再锁定订单行，最后推进支付流水状态。这样可以避免“回调先持有支付流水锁、发起支付先持有订单锁”导致的相互等待。

另外，MySQL RR 隔离级别下普通 `SELECT` 是快照读。对于“唯一键冲突后查询已有流水”或“CAS 失败后返回当前终态”这类幂等返回逻辑，如果必须读取其他事务刚提交的数据，应使用当前读，例如 `SELECT ... FOR UPDATE`，避免返回旧快照。
