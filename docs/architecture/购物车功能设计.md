# 购物车功能设计

## 一、设计目标

本阶段为 `mall-lite` 增加登录用户购物车能力，补齐商品详情与订单确认之间的核心链路。

购物车采用 `Redis + RabbitMQ + MySQL` 架构：

1. Redis 承载用户实时读写。
2. RabbitMQ 异步传递逐项变更事件。
3. MySQL 保存可恢复的持久化数据。
4. Redis 数据缺失时，从 MySQL 回源重建。

本阶段只支持登录用户，不实现游客购物车和登录后的购物车合并。

## 二、业务规则

### 1. 加购与数量

1. 用户必须登录后才能加入购物车。
2. 购物车项以 `userId + skuId` 唯一标识。
3. 重复加入同一个 SKU 时累加数量。
4. 单个 SKU 的数量范围为 `1-99`。
5. 购物车不锁定库存。

### 2. 商品状态与价格

1. 购物车列表展示 SKU 与商品的最新信息。
2. 商品下架或 SKU 下架后，购物车项仍然保留。
3. 失效购物车项显示明确提示，不允许勾选结算。
4. 商品价格变化后，购物车列表和确认订单页展示最新价格。
5. 订单创建时仍由订单模块重新校验商品状态、SKU 状态、实时价格和库存。

### 3. 结算

1. 购物车页支持单选、全选和部分商品结算。
2. 进入确认订单页时不删除购物车项。
3. 订单创建成功后，只删除本次已结算的购物车项。
4. 订单创建失败时，购物车内容保持不变。
5. 购物车清理失败不回滚已创建订单，前端提示用户稍后手动处理。

## 三、模块边界

新增 `mall-cart` 后端模块，负责：

1. 管理登录用户购物车。
2. 以 Redis 承载购物车实时状态。
3. 发送购物车变更事件。
4. 消费变更事件并异步同步 MySQL。
5. Redis 数据不存在时，从 MySQL 回源重建。

订单模块继续只接收 `skuId + quantity` 列表。购物车模块不决定最终成交价，也不直接锁定库存。

`mall-app` 模块负责聚合 `mall-cart`，并放置购物车 Controller、RabbitMQ 配置、生产者和消费者。

## 四、MySQL 数据模型

新增表 `oms_cart_item`：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 主键 |
| `user_id` | `BIGINT UNSIGNED` | 用户 ID |
| `sku_id` | `BIGINT UNSIGNED` | SKU ID |
| `quantity` | `INT UNSIGNED` | 数量，范围 `1-99` |
| `version` | `BIGINT UNSIGNED` | 购物车项版本号，用于处理消息重复和乱序 |
| `create_time` | `DATETIME` | 创建时间 |
| `update_time` | `DATETIME` | 更新时间 |
| `deleted` | `TINYINT UNSIGNED` | 逻辑删除标记：`0` 未删除，`1` 已删除 |

建立唯一索引：

```text
(user_id, sku_id)
```

删除事件不物理删除记录，而是写入更高版本号并标记 `deleted = 1`。即使旧更新消息晚于删除消息到达，也不能让已删除购物车项重新出现。

本阶段不冗余存储 `product_id`。购物车不是订单快照，`product_id` 可以通过 `sku_id` 查询获得。订单明细仍然保留完整快照。

## 五、Redis 数据模型

### 1. 用户购物车

每位用户使用一个 Hash：

```text
mall:cart:user:{userId}
```

Hash 内容：

| 项目 | 内容 |
| --- | --- |
| `field` | `skuId` |
| `value` | 包含 `quantity`、`version`、`deleted` 的 JSON |

删除购物车项时不直接移除 Hash 字段，而是写入更高版本号的墓碑值：

```text
deleted = true
```

查询购物车列表时过滤墓碑值。保留墓碑可以保证删除后再次加购仍继续递增版本号，避免旧消息覆盖新状态。

### 2. 初始化标记

使用单独的初始化标记：

```text
mall:cart:loaded:{userId}
```

该标记用于区分两种情况：

1. 用户购物车确实为空。
2. Redis 尚未从 MySQL 恢复该用户购物车。

## 六、RabbitMQ 变更事件

### 1. 消息粒度

每条消息只描述一个 `userId + skuId` 购物车项的变更。

消息至少包含：

```text
userId
skuId
quantity
version
operation
```

其中 `operation` 分为：

```text
UPSERT
DELETE
```

### 2. 消费幂等与乱序处理

每次 Redis 变更都为购物车项生成更大的版本号。

MySQL 消费者只允许更高版本消息覆盖当前记录：

```text
message.version > oms_cart_item.version
```

重复消息和旧版本消息直接忽略。删除消息也携带版本号，并将数据库记录标记为逻辑删除。

### 3. 生产者确认

生产者发送消息时启用 RabbitMQ Publisher Confirm。只有 Broker 确认接收消息后，购物车写接口才返回成功。

## 七、接口设计

购物车接口统一要求登录态，从 `Authorization` token 解析 `userId`。

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `POST` | `/api/cart/items` | 加入购物车；重复 SKU 累加数量 |
| `GET` | `/api/cart/items` | 查询当前用户购物车 |
| `PUT` | `/api/cart/items/{skuId}` | 修改某个 SKU 的数量 |
| `DELETE` | `/api/cart/items/{skuId}` | 删除单个购物车项 |
| `DELETE` | `/api/cart/items` | 批量删除指定 SKU，供结算成功后调用 |

购物车查询返回最新商品信息，并提供可结算状态：

```text
available = 商品已上架 && SKU 已上架
```

## 八、核心数据流

### 1. 普通写入链路

```text
接收加购、修改或删除请求
→ 校验登录态
→ 校验数量范围 1-99
→ 必要时校验 SKU 存在
→ 使用 Redis Lua 脚本原子更新购物车项并递增 version
→ 发送 RabbitMQ 逐项变更事件
→ 发送失败则使用 Redis Lua 脚本执行基于 version 的条件回滚，并返回失败
→ 消费者按照 version 将最新变更异步同步至 MySQL
```

条件回滚只允许撤销当前请求写入的版本：

```text
Redis 当前 version == 本次写入 version
```

如果 Redis 已经存在更高版本，说明后续请求已成功修改购物车。此时不得使用旧值覆盖新值。

批量删除接口按 SKU 逐项执行删除，并为每个 SKU 发送独立变更事件。

### 2. Redis 回源链路

```text
查询购物车
→ 检查 mall:cart:loaded:{userId}
→ 未初始化：查询 MySQL 全部购物车项并写入 Redis Hash，包含逻辑删除墓碑
→ 设置 loaded 标记
→ 从 Redis 读取并过滤 deleted = true 的墓碑值
→ 批量查询 SKU 与商品最新信息
→ 返回页面展示数据
```

### 3. 购物车结算链路

```text
购物车页勾选有效项
→ 进入确认订单页
→ 确认页携带 skuId + quantity 列表
→ 用户选择地址并提交
→ 复用 POST /api/orders 创建订单
→ 订单创建成功
→ 前端调用购物车批量删除接口
→ 只清理本次已结算 SKU
→ 跳转订单结果页
```

## 九、前端设计

### 1. 商品详情页

“加入购物车”按钮接入真实接口：

```text
未登录 → 跳转登录页，登录后返回商品详情
未选择 SKU → 提示选择规格
已登录且参数合法 → 调用加购接口
成功 → 提示加入购物车成功
```

“立即购买”保留现有行为，不经过购物车。

### 2. 顶部导航

购物车入口改为：

```text
/cart
```

第一版不在顶部导航展示实时购物车数量。

### 3. 购物车页

新增 `/cart` 页面，支持：

1. 展示商品图片、商品名称、SKU 名称、最新价格和数量。
2. 修改数量，范围为 `1-99`。
3. 删除单项。
4. 单选、全选有效项。
5. 失效项保留展示，标记“商品已失效”，禁止勾选。
6. 展示已选数量与合计金额。
7. 将勾选项带入确认订单页。

### 4. 确认订单页

现有确认订单页改为同时支持两种来源：

| 来源 | 参数 | 下单成功后的行为 |
| --- | --- | --- |
| 立即购买 | 单个 `skuId + quantity` | 不清理购物车 |
| 购物车结算 | 多个 `skuId + quantity` | 调用批量删除接口清理已结算项 |

商品展示区支持多行订单明细，金额按全部条目汇总。

## 十、异常处理

1. 加购消息发送失败：基于本次写入版本条件恢复 Redis 原值，接口返回失败。
2. 商品或 SKU 失效：保留购物车项，禁止结算。
3. 价格变化：列表与确认页展示最新价格，下单时订单模块再次计算。
4. 库存不足：购物车不锁库存；下单失败后购物车内容保持不变。
5. 购物车清理失败：不回滚已创建订单，前端提示用户清理失败。
6. Redis 数据缺失：从 MySQL 回源恢复。
7. MQ 重复或乱序：消费者依靠版本号忽略重复消息和旧消息。

## 十一、测试范围

### 1. 后端测试

新增购物车集成测试，覆盖：

1. 登录用户首次加入 SKU。
2. 重复加入同一 SKU 后数量累加。
3. 累加后超过 `99` 时拒绝请求。
4. 修改数量与删除购物车项。
5. 未登录或 token 无效时拒绝访问。
6. Redis 缓存不存在时，从 MySQL 回源重建。
7. 商品或 SKU 下架后，购物车项保留但标记失效。
8. MQ 消费者重复消费同一版本时保持幂等。
9. MQ 消费者收到旧版本时，不覆盖 MySQL 新版本。
10. 删除事件先落库、旧更新事件后到达时，商品仍保持删除状态。
11. MQ 发送失败时，Redis 恢复为修改前状态。

### 2. 前端验证

覆盖以下用户路径：

1. 商品详情页加入购物车。
2. 购物车修改数量、删除商品、单选和全选。
3. 失效商品不可勾选。
4. 勾选多个 SKU 进入确认订单页。
5. 立即购买仍然可以正常下单。
6. 购物车结算成功后，仅清理本次结算项。

## 十二、可靠性边界

本阶段采用：

```text
Redis 主写
→ RabbitMQ 异步同步
→ MySQL 兜底恢复
```

如果 RabbitMQ 已确认接收消息，但消费者尚未写入 MySQL，此时 Redis 同时丢失，购物车查询可能短暂读取到旧数据。MQ 消费完成后，MySQL 才会恢复到最新状态。

第一版接受这个短暂窗口，不额外引入 Outbox、延迟双删或定时对账任务。后续若要提高生产可靠性，可以增加定时校验和失败补偿。

## 十三、本阶段不做

1. 游客购物车。
2. 登录后的游客购物车合并。
3. 顶部导航实时购物车数量。
4. 购物车库存锁定。
5. 优惠券、促销分摊和运费计算。
6. Outbox、定时对账和失败补偿任务。
