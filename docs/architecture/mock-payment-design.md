# 模拟支付设计

## 一、设计目标

当前阶段不接支付宝、微信等真实支付渠道，先用后端模拟第三方支付流程，打通交易主链路：

1. 用户下单后生成待支付订单。
2. 下单时库存从 `available_stock` 转入 `locked_stock`。
3. 用户发起模拟支付，系统创建 `PENDING` 支付流水。
4. 模拟第三方支付通过回调通知成功或失败。
5. 支付成功后，订单状态变为已支付，库存从 `locked_stock` 中确认扣减。
6. 支付失败后，支付流水变为失败，订单仍保持待支付，库存仍保持锁定。

## 二、接口设计

### 1. 发起支付

```text
POST /api/orders/{orderId}/pay
```

发起支付只做两件事：

1. 模拟调用第三方支付预下单接口。
2. 在本地创建或返回已有的待支付流水。

响应核心字段：

```json
{
  "paymentNo": "P202605250830001234567890",
  "orderId": 1,
  "orderNo": "O202605250829001234567890",
  "orderStatus": 10,
  "paymentStatus": 10,
  "payAmount": 398.00
}
```

其中：

1. `orderStatus = 10` 表示订单仍是待支付。
2. `paymentStatus = 10` 表示支付流水待回调确认。
3. 如果同一订单在回调前重复发起支付，返回已有的待支付流水，不重复创建支付流水。

### 2. 支付回调

```text
POST /api/payments/{paymentNo}/callback
```

请求体：

```json
{
  "mockResult": "SUCCESS"
}
```

`mockResult` 当前支持：

1. `SUCCESS`：模拟第三方支付成功。
2. `FAILED`：模拟第三方支付失败。

## 三、核心流程

### 1. 发起支付流程

```text
调用支付接口
-> 模拟第三方支付预下单耗时
-> 开启本地事务
-> 校验订单必须是待支付
-> 查询是否已有 PENDING 支付流水
-> 如果存在，直接返回已有流水
-> 如果不存在，创建 PENDING 支付流水
-> 提交事务
```

### 2. 支付成功回调流程

```text
调用支付回调接口，mockResult = SUCCESS
-> 开启本地事务
-> 查询支付流水
-> 如果支付流水已是终态，直接返回当前结果
-> 校验订单必须是待支付
-> 更新支付流水为 SUCCESS，并写入 pay_time
-> 按订单明细扣减 SKU 锁定库存
-> 更新订单为已支付，并写入 pay_time
-> 提交事务
```

### 3. 支付失败回调流程

```text
调用支付回调接口，mockResult = FAILED
-> 开启本地事务
-> 查询支付流水
-> 如果支付流水已是终态，直接返回当前结果
-> 更新支付流水为 FAILED
-> 订单保持待支付
-> 库存保持锁定
-> 提交事务
```

## 四、回调幂等设计

真实支付渠道可能因为网络抖动、超时重试等原因，多次推送同一笔支付回调。因此回调接口必须具备幂等能力。

当前规则：

1. 只有 `PENDING` 支付流水允许被回调推进状态。
2. `SUCCESS`、`FAILED`、`CLOSED` 都视为终态。
3. 支付流水已进入终态后，后续重复或冲突回调只返回当前状态，不再重复扣库存、不再重复更新订单。

这样可以避免两个风险：

1. 重复成功回调导致 `locked_stock` 被重复扣减。
2. 成功后又收到失败回调，导致订单从已支付被错误改回待支付或失败。

## 五、为什么模拟耗时不放在事务里

代码中 `MockThirdPartyPaymentClient` 使用 `Thread.sleep(300)` 模拟第三方支付渠道耗时，但这个步骤放在本地数据库事务之前。

原因是：真实生产中调用第三方支付属于外部网络 IO，可能慢、可能超时。如果在数据库事务里等待外部接口，会让数据库连接和行锁持有时间变长，降低系统吞吐，也更容易引发锁等待。

当前实现保持这个习惯：

```text
外部耗时操作在事务外
本地状态落库在事务内
```

## 六、当前限制和后续扩展

1. 当前支付回调没有签名验签，真实支付渠道必须校验回调来源和参数签名。
2. 当前支付失败后，订单和库存仍保持待支付与锁定状态，后续可以增加“重新支付”和“超时自动取消释放库存”。
3. 当前回调结果通过 `mockResult` 手动指定，适合 Swagger 和接口测试；后续可以抽象成支付渠道适配器。
4. 当前没有支付超时关闭逻辑，后续可以结合 RabbitMQ 延迟消息或定时任务实现。
