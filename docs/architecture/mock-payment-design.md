# 模拟支付设计

## 一、设计目标

当前阶段不接支付宝、微信等真实支付渠道，先用后端模拟“第三方支付成功”的结果，打通交易主链路：

1. 用户下单后生成待支付订单。
2. 下单时库存从 `available_stock` 转入 `locked_stock`。
3. 用户发起模拟支付。
4. 支付成功后创建支付流水。
5. 订单状态从待支付改为已支付。
6. 库存从 `locked_stock` 中扣减，完成库存确认消耗。

## 二、接口设计

支付接口：

```text
POST /api/orders/{orderId}/pay
```

成功响应核心字段：

```json
{
  "paymentNo": "P202605250830001234567890",
  "orderId": 1,
  "orderNo": "O202605250829001234567890",
  "orderStatus": 20,
  "paymentStatus": 20,
  "payAmount": 398.00
}
```

## 三、核心流程

```text
调用支付接口
-> 模拟第三方支付耗时
-> 开启本地事务
-> 校验订单必须是待支付
-> 创建支付成功流水
-> 扣减订单明细对应 SKU 的锁定库存
-> 更新订单为已支付并写入 pay_time
-> 提交事务
```

## 四、为什么模拟耗时不放在事务里

代码中 `MockThirdPartyPaymentClient` 使用 `Thread.sleep(300)` 模拟第三方支付耗时，但这个步骤放在本地数据库事务之前。

原因是：真实生产中调用第三方支付属于外部网络 IO，可能慢、可能超时。如果在数据库事务里等待外部接口，会让数据库连接和行锁持有时间变长，降低系统吞吐，也更容易引发锁等待。

第一版虽然是模拟支付，也保持这个习惯：

```text
外部耗时操作在事务外
本地状态落库在事务内
```

## 五、当前限制

1. 当前支付结果固定为成功。
2. 当前没有单独的支付回调接口。
3. 当前通过订单状态防止重复支付，后续可以进一步增加支付流水幂等号。
4. 当前没有支付超时关闭逻辑，后续可以结合 RabbitMQ 或定时任务实现。
