# 技术选型说明

## 一、整体选型结论

`mall-lite` 推荐采用 **Spring Boot + MyBatis-Plus + MySQL + Redis + RabbitMQ** 的轻量组合，先以 Maven 多模块单体应用方式落地。

这个组合适合当前阶段的原因是：

1. 启动成本低，便于快速完成电商核心链路。
2. 技术栈贴近 Java 后端主流生产实践。
3. 后续可以自然扩展到 Spring Cloud、消息队列、分布式事务和容器化部署。
4. 对面试复盘友好，能覆盖事务、缓存、并发、数据库设计、接口设计等高频知识点。

## 二、后端技术栈

| 技术 | 版本 | 作用 |
| --- | --- | --- |
| JDK | 21 | 当前主流长期支持版本之一，适合 Spring Boot 3.x 项目 |
| Spring Boot | 3.5.13 | Web 应用、依赖管理、自动配置、统一启动入口 |
| Spring Web | 随 Spring Boot | 提供 REST API 能力 |
| MyBatis-Plus | 3.5.16 | 简化 CRUD、分页、条件构造器和基础 Mapper 编写 |
| MySQL | 8.0 | 存储用户、商品、库存、订单、支付等核心业务数据 |
| Redis | 5.0.14.1 | 缓存热点数据、辅助库存控制、订单超时处理 |
| RabbitMQ | 3.12.0 | 处理订单超时、库存释放、支付通知等异步消息 |
| Maven | 3.9.15 | 多模块项目管理、依赖版本统一控制 |
| Lombok | 可选 | 减少样板代码，后续可根据团队习惯决定是否引入 |

版本说明：

1. Spring Boot 3.5.13 可以配合 JDK 21 使用，项目代码建议统一按 Java 21 编译和运行。
2. Redis 5.0.14.1 对学习项目够用，但版本相对偏旧，后续如果做容器化或生产化演示，可以再升级到 Redis 7.x 或更新版本。
3. RabbitMQ 3.12.0 可以用于学习 AMQP、延迟队列、订单超时和异步解耦；如果后续追求稳定性，建议升级到同一 3.12 系列的较新补丁版本。

## 三、为什么先不用完整微服务

### 1. 当前阶段优先目标

轻量电商系统的第一阶段重点是把业务主流程跑通，而不是一开始就引入注册中心、网关、配置中心、链路追踪等基础设施。

先做单体多模块有几个好处：

1. 模块边界清楚，但部署复杂度低。
2. 调试链路短，适合学习业务流程。
3. 可以先把事务、库存、订单状态流转这些核心问题讲明白。
4. 后续拆服务时，`mall-user`、`mall-product`、`mall-order` 等模块可以作为天然边界。

### 2. 后续可扩展方向

| 扩展方向 | 可引入技术 | 典型用途 |
| --- | --- | --- |
| 服务拆分 | Spring Cloud Alibaba | 用户、商品、订单、库存、支付独立部署 |
| 服务注册 | Nacos | 服务发现和配置管理 |
| API 网关 | Spring Cloud Gateway | 统一鉴权、限流、路由 |
| 异步解耦 | RabbitMQ | 订单超时、库存释放、支付通知 |
| 分布式事务 | Seata 或可靠消息最终一致性 | 订单、库存、支付跨服务一致性 |
| 链路追踪 | Micrometer Tracing | 排查跨服务调用问题 |

## 四、数据库、缓存与消息队列选型

### 1. MySQL

MySQL 负责存储强一致性的核心业务数据，例如：

1. 用户信息。
2. 商品和 SKU 信息。
3. 库存数量和锁定库存。
4. 订单主表和订单明细。
5. 支付流水。

生产实践中，订单和库存相关表要重点关注：

1. 事务边界。
2. 行锁竞争。
3. 唯一索引和幂等约束。
4. 金额字段精度。
5. 状态机设计。

### 2. Redis

Redis 前期可以先用于三个场景：

1. 缓存商品详情和分类数据，减少数据库查询压力。
2. 保存登录态或验证码等短期数据。
3. 辅助处理订单超时和库存预热。

需要注意的是，Redis 不应该替代 MySQL 成为订单和支付的最终事实来源。订单、支付、库存最终状态仍然要以数据库为准。

### 3. RabbitMQ

RabbitMQ 前期可以先用于三个场景：

1. 订单超时未支付后发送取消订单消息。
2. 订单取消后异步释放库存。
3. 支付成功后异步通知订单模块更新状态。

生产实践中，使用 RabbitMQ 要重点关注：

1. 消息幂等，避免重复消费导致库存或订单状态异常。
2. 消息确认机制，避免消息丢失。
3. 死信队列，处理超时订单和消费失败消息。
4. 重试策略，避免无限重试拖垮系统。
5. 业务最终一致性，订单、库存、支付之间不能只依赖内存状态。

## 五、模块依赖建议

推荐的依赖方向如下：

```text
mall-app
├── mall-user
├── mall-product
├── mall-inventory
├── mall-order
├── mall-payment
└── mall-common

mall-user       -> mall-common
mall-product    -> mall-common
mall-inventory  -> mall-common
mall-order      -> mall-common, mall-user, mall-product, mall-inventory
mall-payment    -> mall-common, mall-order
```

这个依赖关系体现了两个原则：

1. `mall-common` 只放通用能力，不反向依赖业务模块。
2. 订单是核心编排模块，可以调用用户、商品、库存能力完成下单流程。

## 六、接口风格

接口建议采用 REST 风格，并统一响应结构。

示例：

```text
POST   /api/orders              创建订单
GET    /api/orders/{orderId}    查询订单详情
POST   /api/orders/{orderId}/cancel
POST   /api/payments/pay
GET    /api/products/{skuId}
```

统一响应对象后续可以放在 `mall-common` 中，例如：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

## 七、开发阶段路线

### 1. 第一阶段：基础骨架

1. 创建 Maven 多模块项目。
2. 配置 Spring Boot 父工程。
3. 创建 `mall-app` 启动类。
4. 建立统一返回、异常处理、基础枚举。

### 2. 第二阶段：数据库与基础业务

1. 设计用户、商品、库存、订单、支付表。
2. 使用 MyBatis-Plus 完成基础 CRUD。
3. 打通商品查询、库存查询、订单创建流程。

### 3. 第三阶段：核心交易链路

1. 实现库存锁定。
2. 实现订单状态机。
3. 实现模拟支付。
4. 实现支付成功后扣减库存。
5. 实现订单取消后释放库存。

### 4. 第四阶段：工程增强

1. 引入 Redis 缓存。
2. 引入 RabbitMQ 处理订单超时、库存释放和支付通知。
3. 增加参数校验和全局异常处理。
4. 补充接口文档。
5. 增加单元测试和集成测试。
6. 编写 Docker Compose 启动配置。

## 八、面试表达建议

如果面试中介绍这个项目，可以这样表达：

> 这个项目是一个轻量级电商系统，采用 Spring Boot 单体多模块架构。前期没有直接上微服务，而是先通过 Maven 模块划分用户、商品、库存、订单、支付等业务边界，重点实现下单、库存锁定、支付回调、订单取消和库存释放这些核心链路。这样既能降低开发复杂度，也为后续拆分 Spring Cloud 微服务保留了清晰边界。

这个表达能体现三点：

1. 你理解业务核心链路。
2. 你没有为了技术而技术，知道按阶段控制复杂度。
3. 你具备后续演进到微服务的架构意识。
