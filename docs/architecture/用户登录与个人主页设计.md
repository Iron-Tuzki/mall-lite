# 用户登录与个人主页设计

## 一、设计目标

当前前台已经具备商品浏览、商品详情、分类页和下单相关能力，下一步需要把用户身份真正接入前端页面。第一版目标是：

1. 用户可以在前台登录。
2. 登录成功后，前端保存 token 和当前用户信息。
3. 后续接口通过 `Authorization: Bearer <token>` 携带登录态。
4. 用户可以进入个人主页查看自己的基础信息和常用入口。
5. 退出登录后，前端清理登录态，后端删除 Redis 中的 token。

## 二、后端现有接口检查

### 1. 登录接口

```text
POST /api/users/login
```

请求体：

```json
{
  "username": "test",
  "password": "password123"
}
```

返回数据：

```json
{
  "token": "random-token",
  "user": {
    "id": 1,
    "username": "test",
    "nickname": "测试用户",
    "phone": "13800138000",
    "email": "test@example.com",
    "avatarUrl": null,
    "status": 1
  }
}
```

### 2. 当前用户接口

```text
GET /api/users/me
Authorization: Bearer <token>
```

作用：前端刷新页面后，可以用本地 token 重新拉取当前用户信息。

### 3. 退出登录接口

```text
POST /api/users/logout
Authorization: Bearer <token>
```

作用：服务端删除 Redis 中的登录 token，使当前登录态立即失效。

### 4. 结论

后端登录和当前用户接口已经满足第一版个人主页需要，不需要新增后端接口。个人主页第一版先使用 `GET /api/users/me` 展示用户基础信息，订单统计、最近订单和收货地址入口后续再接真实接口。

## 三、前端登录态设计

### 1. 状态存储

前端使用 Pinia 新增 `auth` store，负责保存：

1. `token`
2. `user`
3. `isLoggedIn`
4. 登录、拉取当前用户、退出登录等动作

token 持久化到 `localStorage`，页面刷新后仍能恢复。

### 2. 请求头处理

Axios 请求拦截器从统一的 token 存储工具读取 token，然后自动附加：

```text
Authorization: Bearer <token>
```

这样商品、订单、地址、支付等接口不需要每个方法手动传 token。

### 3. 页面路由

新增页面：

```text
/login    登录页
/profile  个人主页
```

`/profile` 需要登录。如果没有 token，跳转到：

```text
/login?redirect=/profile
```

登录成功后跳回原来的目标页。

## 四、个人主页第一版布局

个人主页先做成轻量客户中心，展示：

1. 用户信息：头像、昵称、用户名、手机号脱敏、邮箱。
2. 订单快捷入口：待支付、待发货、待收货、已完成、已取消。
3. 最近订单占位：后续接入我的订单接口。
4. 我的服务：收货地址、账号安全、退出登录。

第一版重点是登录态闭环，不急着把所有业务数据都接满。

## 五、后续扩展

1. 接入 `GET /api/orders/my/statistics` 返回订单状态数量。
2. 接入 `GET /api/orders/my?pageNo=1&pageSize=5` 展示最近订单。
3. 接入地址接口，在个人主页展示默认收货地址。
4. 增加注册页，或者在登录页提供跳转注册入口。
5. 后续如果接入 Spring Security，可以把当前手写 token 解析迁移到统一过滤器。
