# 下单接口、幂等与库存锁定设计

## 一、设计目标

下单链路需要同时解决三个核心问题：

1. **多 SKU 下单**：一张订单可以包含多条订单明细，关系为 `oms_order 1 : N oms_order_item`。
2. **请求幂等**：同一个登录用户使用同一个 `requestId` 重复提交时，只能创建一张订单。
3. **库存安全**：并发下不能超卖，锁库存成功后订单、订单明细和库存变化要保持一致。

当前版本暂不引入购物车、优惠券、运费、多仓库存、Redis 预扣库存、分布式事务和秒杀削峰，只打通普通下单场景下的核心交易链路。

## 二、接口边界

### 1. 用户下单接口

```text
POST /api/orders
```

请求头：

```text
Authorization: Bearer <token>
```

请求体：

```json
{
  "requestId": "b7d4f6c8-xxxx-xxxx",
  "addressId": 900001,
  "items": [
    {
      "skuId": 900001,
      "quantity": 2
    },
    {
      "skuId": 900002,
      "quantity": 1
    }
  ],
  "remark": "please ship soon"
}
```

字段说明：

1. `requestId`：下单请求幂等号，由前端在用户提交订单前生成。
2. `addressId`：当前登录用户名下的收货地址 ID。
3. `items`：订单明细列表，一条记录表示一个 SKU 及其购买数量。
4. `remark`：订单备注，可为空。

约束规则：

1. `userId` 不由前端传入，后端从 `Authorization` 登录态中解析。
2. `requestId` 不能为空，同一用户下必须唯一表达一次下单尝试。
3. `items` 不能为空。
4. 每个明细必须包含 `skuId` 和大于 0 的 `quantity`。
5. 同一个请求内不允许重复传入相同 `skuId`，第一版不在后端自动合并数量。
6. 不同用户可以使用相同 `requestId`，互不影响。

### 2. 库存管理接口

后台库存管理接口只负责维护库存基础数据：

```text
POST   /api/admin/inventories
GET    /api/admin/inventories/{skuId}
PUT    /api/admin/inventories/{skuId}
DELETE /api/admin/inventories/{skuId}
```

用户下单时的锁库存不单独暴露为用户侧接口，而是由订单模块在 `POST /api/orders` 内部编排库存模块能力：

```text
OrderController
  -> OrderService.createOrder(...)
      -> 校验登录态和收货地址
      -> 校验请求幂等
      -> 校验 SKU 和商品状态
      -> 调用 InventoryService.lockStock(...)
      -> 创建订单主表
      -> 创建订单明细表
      -> 返回订单结果
```

锁库存是下单流程中的内部步骤，不是一个独立的用户动作。

## 三、核心数据设计

### 1. 订单请求幂等表

为了避免并发重复请求都进入锁库存逻辑，新增 `oms_order_request` 作为下单请求幂等表，把 `requestId` 抢占提前到库存锁定之前。

```sql
CREATE TABLE oms_order_request
(
    id          BIGINT UNSIGNED  NOT NULL AUTO_INCREMENT COMMENT '订单请求ID',
    user_id     BIGINT UNSIGNED  NOT NULL COMMENT '用户ID',
    request_id  VARCHAR(64)      NOT NULL COMMENT '下单请求幂等号',
    order_id    BIGINT UNSIGNED  NULL COMMENT '创建成功后的订单ID',
    status      TINYINT UNSIGNED NOT NULL DEFAULT 10 COMMENT '请求状态：10处理中，20成功',
    create_time DATETIME         NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME         NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    deleted     TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '逻辑删除标记：0未删除，1已删除',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_request (user_id, request_id),
    KEY idx_order_id (order_id)
) COMMENT = '订单请求幂等表';
```

字段含义：

1. `user_id`：当前登录用户 ID。
2. `request_id`：前端生成的下单请求幂等号。
3. `order_id`：下单成功后关联的订单 ID。
4. `status`：请求状态，`10` 表示处理中，`20` 表示已成功创建订单。

### 2. 订单表保留幂等号

`oms_order` 继续保留 `request_id`：

```sql
request_id VARCHAR(64) NULL COMMENT '下单请求幂等号'
```

并保留唯一索引：

```sql
UNIQUE KEY uk_user_request (user_id, request_id)
```

`oms_order_request` 负责入口抢占，`oms_order` 的唯一索引作为最终兜底约束。

### 3. 库存表核心字段

`ims_inventory` 第一版使用两个核心库存字段：

| 字段 | 含义 | 变化场景 |
| --- | --- | --- |
| `available_stock` | 可用库存 | 下单锁库存时减少，取消订单释放库存时增加 |
| `locked_stock` | 锁定库存 | 下单锁库存时增加，支付成功扣减库存或取消订单释放库存时减少 |

`version` 字段可以保留，用于观察库存变更次数，也为后续后台库存调整、独立库存服务或乐观锁扩展预留空间。

## 四、下单主流程

```text
接收下单请求
-> 校验 Authorization
-> 从 Redis 登录态读取当前 userId
-> 根据 userId + requestId 查询是否已有成功订单
-> 如果已有订单，直接返回第一次创建的订单
-> 插入 oms_order_request，状态为 PROCESSING
-> 如果插入失败，说明重复请求已被其他线程抢占
   -> 短暂等待并重查已有订单
   -> 如果订单已创建，返回原订单
   -> 如果多次重查后订单仍不可见，返回“请求处理中”
-> 校验收货地址属于当前用户
-> 校验 items 非空、数量合法、无重复 skuId
-> 查询并校验 SKU、商品状态
-> 逐个锁定库存
-> 汇总订单金额
-> 创建 oms_order，写入 userId 和 requestId
-> 创建多条 oms_order_item
-> 更新 oms_order_request 为 SUCCESS，并写入 order_id
-> 返回订单创建结果
```

关键规则：

1. **只有成功插入 `oms_order_request` 的请求，才允许继续锁库存。**
2. 并发重复请求会被 `oms_order_request.uk_user_request` 拦截，不会重复进入库存锁定逻辑。
3. `oms_order.uk_user_request` 保留为兜底，防止异常路径下重复订单落库。
4. 多 SKU 下单必须在同一个本地数据库事务中完成，任意一个 SKU 失败则整体回滚。

## 五、库存锁定规则

### 1. 下单锁库存

用户提交订单并通过基础校验后，库存变化为：

```text
available_stock = available_stock - quantity
locked_stock    = locked_stock + quantity
```

示例：

```text
下单前：available_stock = 100，locked_stock = 0
购买 2 件后：available_stock = 98，locked_stock = 2
```

### 2. 取消订单释放库存

未支付订单取消时，库存变化为：

```text
available_stock = available_stock + quantity
locked_stock    = locked_stock - quantity
```

示例：

```text
取消前：available_stock = 98，locked_stock = 2
释放后：available_stock = 100，locked_stock = 0
```

### 3. 支付成功扣减锁定库存

订单支付成功后，锁定库存真正出库：

```text
locked_stock = locked_stock - quantity
```

支付成功时不再减少 `available_stock`，因为可用库存已经在下单锁库存阶段减少过。

示例：

```text
支付前：available_stock = 98，locked_stock = 2
支付后：available_stock = 98，locked_stock = 0
```

### 4. 条件更新防超卖

不推荐使用“先查询库存，再无条件更新库存”的方式：

```text
1. SELECT available_stock FROM ims_inventory WHERE sku_id = ?
2. Java 判断库存是否足够
3. UPDATE ims_inventory SET available_stock = available_stock - ?
```

这种方式在并发下容易超卖。第一版推荐使用 MySQL 条件更新，把“判断库存是否足够”和“锁定库存”合并为一条原子 SQL：

```sql
UPDATE ims_inventory
SET available_stock = available_stock - #{quantity},
    locked_stock = locked_stock + #{quantity},
    version = version + 1,
    update_time = NOW()
WHERE sku_id = #{skuId}
  AND deleted = 0
  AND available_stock >= #{quantity}
```

执行结果判断：

1. 影响行数为 `1`：锁库存成功。
2. 影响行数为 `0`：库存不足、SKU 库存不存在，或库存记录已删除。

条件更新依赖 MySQL 单行更新的原子性，可以避免并发超卖。

## 六、事务和隔离级别

### 1. 本地事务边界

下单事务必须覆盖：

1. 锁定库存。
2. 创建订单主表。
3. 创建订单明细。
4. 更新 `oms_order_request` 成功状态。

事务规则：

1. 任意 SKU 库存不足，整个下单失败。
2. 任意订单明细创建失败，已锁库存随事务回滚。
3. 订单主表、订单明细、库存锁定和幂等请求状态要一起提交或一起回滚。

如果库存锁定成功后订单创建失败但没有事务回滚，就会出现库存被长期占用的问题。

### 2. 重复请求读取已创建订单

由于 MySQL 默认隔离级别是可重复读，如果重复请求在同一个事务中反复查询，可能一直读到旧快照，看不到另一个事务刚提交的订单。

因此下单事务建议使用 `READ_COMMITTED`，让重复请求在短暂等待后可以读取到原请求已经提交的订单结果。

## 七、金额与订单明细快照

订单总金额来自所有明细小计之和：

```text
order.total_amount = sum(sku.price * quantity)
order.pay_amount = order.total_amount
```

当前版本没有优惠券、促销和运费，所以 `pay_amount` 暂时等于 `total_amount`。

订单明细保存下单时快照，避免后续商品信息变化影响历史订单展示：

1. 商品 ID、SKU ID。
2. 商品名称、SKU 名称、SKU 编码。
3. 规格、主图。
4. 下单时单价、购买数量、明细小计。

## 八、错误处理

| 场景 | code | message |
| --- | --- | --- |
| 未传 `Authorization` | `401` | `missing login token` |
| token 为空 | `401` | `missing login token` |
| token 不存在或已退出 | `401` | `invalid login token` |
| 收货地址不属于当前用户 | `404` | `address not found` |
| `items` 为空 | `400` | `items must not be empty` |
| 购买数量小于 1 | `400` | `quantity must be greater than 0` |
| 同一请求内重复传入 SKU | `400` | `duplicate sku in order items` |
| SKU 不存在或未上架 | `404` | `sku not found` |
| 库存记录不存在 | `404` | `inventory not found` |
| 库存不足 | `400` | `insufficient stock` |
| 重复请求命中处理中记录但订单尚未可见 | `409` | `order request is processing` |
| 订单创建失败 | `500` 或业务异常 | `create order failed` |

条件更新影响行数为 `0` 时，可以再查询库存记录辅助区分 `inventory not found` 和 `insufficient stock`，让接口提示更清晰。

## 九、和其他幂等或一致性能力的区别

| 能力 | 解决的问题 | 当前实现 |
| --- | --- | --- |
| 防超卖 | 多个请求不能把库存扣成负数 | `ims_inventory` 条件更新 |
| 下单幂等 | 同一个下单请求不能创建多张订单，也不能重复锁库存 | `oms_order_request` 入口抢占 + `oms_order` 唯一索引兜底 |
| 支付回调幂等 | 同一支付结果不能重复扣减锁定库存 | `payment_no + status` 条件更新 |
| 超时取消一致性 | 未支付订单到期后释放锁定库存 | 订单状态校验 + 库存释放 |

## 十、代码结构建议

### 1. 库存模块

建议在 `mall-inventory` 中提供库存业务方法：

```text
InventoryService.lockStock(skuId, quantity)
InventoryService.releaseStock(skuId, quantity)
InventoryService.deductLockedStock(skuId, quantity)
```

其中下单第一阶段必须实现 `lockStock`，取消订单和支付成功链路分别使用 `releaseStock`、`deductLockedStock`。

### 2. Mapper 层

建议在 `InventoryMapper` 中提供条件更新方法：

```text
int lockStock(Long skuId, Integer quantity)
```

当前项目还没有 XML Mapper 时，第一版可以使用注解 SQL，保持实现轻量。

### 3. 订单模块

订单模块围绕以下结构实现：

```text
Order
OrderItem
OrderRequest
OrderMapper
OrderItemMapper
OrderCreateRequest
OrderService
OrderController
```

订单创建时由 `OrderService` 统一编排幂等抢占、用户与地址校验、SKU 校验、锁库存、订单落库和订单明细落库。

## 十一、代码注释约定

实现复杂逻辑时，代码注释遵循以下规则：

1. 类和接口继续使用中文 Java 标准注释说明职责。
2. 不给简单 getter、setter、普通赋值写无意义注释。
3. 对并发、事务、状态流转、库存数量变化写关键注释。
4. 注释重点解释“为什么这样做”，而不是重复“这行代码做了什么”。

锁库存方法中可以说明：

```java
// 使用条件更新把“校验库存”和“锁定库存”合并为一次原子操作，避免并发下先查后改导致超卖。
```

订单创建方法中可以说明：

```java
// 锁库存、创建订单、创建订单明细必须在同一个事务中完成，避免库存已锁定但订单创建失败。
```

幂等抢占方法中可以说明：

```java
// 先抢占下单请求幂等记录，只有抢占成功的请求才能继续锁库存，避免重复请求重复占用库存。
```

## 十二、后续扩展方向

### 1. 优惠、运费和促销分摊

后续加入购物车、优惠券、促销、积分或运费后，可以在 `items` 和订单金额字段上继续扩展价格来源、优惠明细和分摊结果。

### 2. 订单取消释放库存

订单取消时调用：

```text
InventoryService.releaseStock(skuId, quantity)
```

只允许待支付订单释放库存，避免已支付订单或已取消订单重复释放。

### 3. 支付成功扣减锁定库存

支付成功时调用：

```text
InventoryService.deductLockedStock(skuId, quantity)
```

只减少锁定库存，不再减少可用库存。

### 4. RabbitMQ 超时关单

订单创建成功后可以发送延迟消息：

```text
订单创建成功
  -> 发送订单超时检查消息
  -> 到期后检查订单是否仍为待支付
  -> 如果未支付，则取消订单并释放库存
```

### 5. 库存流水表

当库存逻辑变复杂后，可以新增库存流水表记录每次库存变化：

```text
sku_id
order_id
change_type
change_quantity
before_available_stock
after_available_stock
before_locked_stock
after_locked_stock
```

库存流水有助于排查库存异常、重复释放、重复扣减等生产问题。

### 6. 拆分微服务后的最终一致性

当前项目是单体多模块，可以用本地事务保证一致性。后续如果拆成订单服务和库存服务，就不能直接依赖本地事务，需要考虑：

1. 可靠消息最终一致性。
2. 分布式事务框架。
3. 库存冻结流水表。
4. 订单状态补偿任务。

第一版先不引入这些复杂方案。
